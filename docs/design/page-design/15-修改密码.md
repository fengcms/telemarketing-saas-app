## 1. 页面概述

| 项目 | 说明 |
|------|------|
| **页面定位** | 用户修改登录密码的表单页面，需验证旧密码后设置新密码 |
| **目标用户** | 所有已登录用户 |
| **核心目标** | 安全地修改用户登录密码，确保旧密码验证通过且新密码符合强度要求 |
| **页面路由** | `/mine/change-password` |
| **入口** | 设置页 → 修改密码 |
| **注意** | 与强制改密不同，正常改密需要提供旧密码 |

## 4. 交互流程

### 4.1 密码显示/隐藏切换

```
用户点击 TDInput 右侧眼睛图标
  → 图标切换（眼睛 ↔ 眼睛划线）
  → 输入框内容切换（密文 ↔ 明文）
  → 三个输入框独立控制，互不影响
```

### 4.2 密码强度实时检测

```
用户在新密码 TDInput 中输入内容
  → 每次输入变化时实时计算密码强度
  → 更新进度条填充段数和颜色
  → 更新强度文字标签
  → 输入为空时隐藏强度指示器
```

### 4.3 表单校验（提交时）

```
用户点击「确认修改」TDButton
  → 前端按顺序校验：
    1. 旧密码为空 → 旧密码 TDInput 下方显示 "请输入旧密码"（`error-7` 色，`fontBodyExtraSmall`）
    2. 新密码为空 → 新密码 TDInput 下方显示 "请输入新密码"
    3. 新密码不足 8 位 → 新密码 TDInput 下方显示 "密码至少 8 位"
    4. 新密码与旧密码相同 → 新密码 TDInput 下方显示 "新密码不能与旧密码相同"（前端前置校验，后端不校验此项）
    5. 确认新密码为空 → 确认新密码 TDInput 下方显示 "请再次输入新密码"
    6. 确认新密码 ≠ 新密码 → 确认新密码 TDInput 下方显示 "两次输入的密码不一致"
  → 任一校验失败 → 对应 TDInput 边框变 `error-7` 色，滚动到第一个错误位置
  → 全部通过 → 进入提交流程
```

### 4.4 提交修改

```
表单校验通过
  → TDButton 进入 Loading 态（显示 TDLoading，禁止点击）
  → POST /api/auth/change-password { oldPassword, newPassword }
  → 成功响应：
      → 按钮恢复正常态
      → **强制重新登录（无例外）**：后端改密成功后自增 tokenVersion，
        所有旧 token（含当前会话）立即失效，旧 token 的后续请求必然 401
      → 清除本地 Token + 用户缓存
      → `TDToast` 提示 "密码已修改，请重新登录"
      → 跳转登录页（清除任务栈）
  → 失败响应（旧密码错误，error.code = VALIDATION）：
      → 按钮恢复正常态
      → 旧密码 TDInput 下方显示 "旧密码不正确"（`error-7` 色）
      → 旧密码 TDInput 边框变 `error-7` 色
  → 失败响应（新密码不足 8 位等参数校验失败，error.code = VALIDATION）：
      → 按钮恢复正常态
      → 新密码 TDInput 下方显示对应错误信息（`error-7` 色）
  → 失败响应（401 Token 过期）：
      → 跳转登录页
  → 网络错误：
      → 按钮恢复正常态
      → `TDToast` 提示 "网络错误，请重试"
```

### 4.5 输入框错误清除

```
用户在显示错误的 TDInput 中重新输入
  → 清除该 TDInput 的错误提示文字
  → 恢复 TDInput 边框为正常样式
```

## 6. 接口依赖

### 6.1 修改密码

> 响应信封、错误码等统一遵循 [00-全局API约定](./00-全局API约定.md)。

```
POST /api/auth/change-password

请求头：
  Authorization: Bearer {token}

请求体（JSON）：
{
  "oldPassword": "string",     // 旧密码（明文，HTTPS 传输）
  "newPassword": "string"      // 新密码（明文，HTTPS 传输，至少 8 位）
}

成功响应（200，{success, data, error} 信封）：
{
  "success": true,
  "data": { "success": true },
  "error": null
}

注意：改密成功后后端自增 tokenVersion，所有旧 token（含当前 access/refresh
token）立即失效 → 前端必须清除本地 Token + 用户缓存并跳转登录页（强制重新登录）。

失败响应（{success, data, error} 信封，错误码以 00-全局API约定 §8 为准）：
| HTTP 状态码 | error.code   | 含义                          | 前端处理 |
|-------------|--------------|------------------------------|---------|
| 400         | VALIDATION   | 旧密码不正确 / 新密码不足 8 位等 | 对应 TDInput 显示错误 |
| 401         | —            | Token 过期                    | 跳转登录页 |
| 500         | —            | 服务器错误                    | `TDToast` 提示 |

说明：
- "新密码与旧密码相同" 后端不校验，由前端前置拦截（见 4.3）
- 后端不存在 1001/1002/1003 等数字业务码，一律使用字符串错误码
```

## 7. 边界情况与备注

| 场景 | 处理方式 |
|------|---------|
| 密码包含空格 | 允许，不做 trim 处理（密码中的空格是有效字符） |
| 粘贴密码 | 允许粘贴，密码强度实时重新计算 |
| 输入法自动大写 | 旧密码和确认密码框禁用自动大写；新密码框允许（用户可能有意大写） |
| 键盘类型 | 默认显示密码键盘（含特殊字符） |
| 快速连续点击确认 | TDButton Loading 态防止重复提交 |
| 页面返回 | 若有输入内容，弹出 `TDDialog` 确认 "密码未修改，确定返回？" |
| 密码输入框自动填充 | 禁用系统自动填充：Flutter 写法为不设置 `autofillHints`（或显式置空），并将三个输入框置于独立 `AutofillGroup` / `AutofillScope` 中、提交后调用 `AutofillGroup.of(context)?.dispose()`，避免密码被自动填充带来安全风险（不使用 Android 原生 `importantForAutofill` 属性） |
| 屏幕旋转 | 保持表单状态，输入内容不丢失 |
| 弱密码是否允许提交 | MVP 阶段仅做提示，不强制拦截（强度为"弱"时仍可提交） |
| 接口超时 | 15s 超时，超时后 `TDToast` 提示 "请求超时，请重试" |
