## 1. 页面概述

### 1.1 页面定位

强制改密页是电销工作台 APP 的安全认证页面。当管理员重置了用户密码后，用户首次登录时系统检测到 `user.mustResetPassword = 1`，将自动跳转至本页面，要求用户设置新密码后方可正常使用 APP。

此外，即使用户绕过了登录时的检测（理论上不应发生），APP 网络层全局拦截器也会捕获后续 API 请求中的 `423 FORCE_CHANGE_PASSWORD` 状态码，作为兜底机制将用户引导至本页面。

### 1.2 目标用户

被管理员重置过密码的电销工作台用户。

### 1.3 核心目标

- 引导用户设置符合安全要求的新密码（至少 8 位）
- 通过密码强度指示器帮助用户创建强密码
- 调用改密接口完成密码更新
- 成功后清除本地 Token，引导用户返回登录页使用新密码重新登录
- 全局拦截 `423 FORCE_CHANGE_PASSWORD`，确保未改密用户无法绕过此流程

### 1.4 触发条件

| 触发路径 | 说明 |
|---------|------|
| **主路径（首选）** | 登录成功 → 检测 `user.mustResetPassword == 1` → 跳转改密页 |
| **兜底路径** | 任意 API 请求返回 `423 FORCE_CHANGE_PASSWORD` → 网络拦截器捕获 → 不走 refresh→retry 路径 → 直接跳转改密页 |

## 4. 交互流程

### 4.1 主流程（正常改密）

```
用户操作                          系统响应
─────────────────────────────────────────────────────────────
1. 从登录页跳转至改密页            → 页面加载，初始化组件
                                  → 焦点默认在新密码输入框

2. 输入新密码                     → 实时更新密码强度指示器
                                  → 实时最低强度提示：长度 < 8 或未同时含字母和数字时，
                                    新密码框下方提示"密码至少 8 位，且需包含字母和数字"
                                  → 若确认密码框已有内容，实时校验一致性

3. 输入确认密码                   → 实时校验与新密码的一致性
                                  → 不一致时确认密码框边框变为 error-7，下方显示提示
                                  → 一致时清除错误状态

4. 点击 [确认] 按钮              → 收起键盘
                                  → 前端表单校验：
                                    ├─ 新密码为空 → 新密码框提示"请输入新密码"
                                    ├─ 新密码 < 8 位 → 新密码框提示"密码至少 8 位"
                                    ├─ 新密码未同时含字母和数字 → 新密码框提示"密码需同时包含字母和数字"
                                    ├─ 确认密码为空 → 确认密码框提示"请再次输入新密码"
                                    └─ 两次密码不一致 → 确认密码框提示"两次密码输入不一致"
                                  → 校验通过：
                                    ├─ 按钮进入 loading 态
                                    ├─ 禁用所有输入框与按钮
                                    └─ 发起 POST /api/auth/change-password

                                  → 成功响应 (200)：
                                    ├─ 按钮恢复默认态
                                    ├─ 显示 TDToast："密码修改成功，请重新登录"
                                    │   （TDToast 样式：底部居中，时长 LONG）
                                    ├─ 清空本地存储的 accessToken
                                    ├─ 清空本地存储的 refreshToken
                                    ├─ 清空本地用户缓存
                                    ├─ 延迟 1.5s 后跳转至 [登录页]
                                    └─ 页面转场动画 (页面 pop：向右侧滑出，250ms easeInOut)

                                  → 失败响应 (400, VALIDATION)：密码不符合强度/与原密码相同
                                    ├─ 按钮恢复默认态
                                    ├─ 恢复输入框可编辑（保留已输入内容）
                                    └─ 错误提示区显示服务端返回的错误信息（可修改后重试，不离开本页）

                                  → 失败响应 (401)：Token 失效
                                    ├─ 清空本地 Token
                                    └─ 跳转 [登录页] 重新登录

                                  → 失败响应 (423, FORCE_CHANGE_PASSWORD)：
                                    │  （理论上不应在此页面收到此状态码，
                                    │    但作为防御性编程处理）
                                    ├─ 按钮恢复默认态
                                    └─ 保持当前页面（已在改密页）

                                  → 网络异常 / 超时：
                                    ├─ 按钮恢复默认态
                                    ├─ 恢复输入框可编辑
                                    └─ 错误提示区显示："网络连接异常，请检查网络后重试"
```

### 4.2 兜底触发流程（全局拦截）

```
任意页面发起 API 请求
  → 响应 423 (FORCE_CHANGE_PASSWORD)
  → 网络层全局拦截器捕获
  → 判断：是否为 refresh 请求（POST /api/auth/refresh）？
    ├─ 是 → 不走 refresh→retry 路径（避免死循环）
    │       → 清空本地 Token
    │       → 跳转 [强制改密页]
    └─ 否 → 不走 refresh→retry 路径
            → 跳转 [强制改密页]（携带当前 accessToken 用于改密请求）
```

### 4.3 返回按钮处理

```
用户点击 TDNavBar 返回按钮
  → 弹出 TDDialog 确认框：
    ┌────────────────────────────┐
    │     退出确认               │
    │                            │
    │  修改密码前无法使用系统     │
    │  功能，确定要退出吗？       │
    │                            │
    │   [取消]     [确定退出]     │
    └────────────────────────────┘

  → 点击 [取消] → 关闭 TDDialog，留在改密页
  → 点击 [确定退出] → 清空本地 Token → 跳转 [登录页]
```

## 6. 接口依赖

### 6.1 修改密码接口

| 项目 | 说明 |
|------|------|
| **URL** | `POST /api/auth/change-password` |
| **Content-Type** | `application/json` |
| **认证** | 需要 `Authorization: Bearer {accessToken}` |

> **重要说明**：此接口为强制改密场景，管理员已重置用户密码，因此 **不需要传入 `oldPassword`**。仅需传入新密码即可。

**请求参数：**

```json
{
  "newPassword": "NewP@ssw0rd"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `newPassword` | `string` | 是 | 新密码，最少 8 位 |

**成功响应 (200)：**（统一信封 `{ success, data, error }`，见 [00-全局API约定](./00-全局API约定.md) §1）

```json
{
  "success": true,
  "data": { "success": true },
  "error": null
}
```

> **后端行为**：成功后端自增 `tv`（Token Version），使该用户所有已签发的 Token 立即失效。

**失败响应：**（统一信封 `{ success: false, data: null, error: { code, message } }`，见 [00-全局API约定](./00-全局API约定.md) §1/§8）

```json
{ "success": false, "data": null, "error": { "code": "VALIDATION", "message": "password too short" } }
```

**错误处理按 HTTP 状态拆分为两类：**

| HTTP | code | 含义 | 客户端处理 |
|------|------|------|-----------|
| `400` | `VALIDATION` | 密码不符合强度（长度不足/未含字母+数字）或与原密码相同 | **页面内提示，可重试**：错误提示区显示对应文案，恢复输入框可编辑并保留已输入内容，用户修改后可再次提交，不离开本页 |
| `401` | `AUTH_EXPIRED` / `AUTH_INVALID` | Token 已失效（如改密成功后 `tv` 自增使旧 Token 作废） | **跳回登录页重新登录**：清空本地 Token → 跳转 [登录页] |
| `423` | `FORCE_CHANGE_PASSWORD` | 仍需强制改密（防御性处理） | 保持当前页面，不跳转 |
| `500` | `INTERNAL` | 服务端异常 | 错误提示区显示"服务器异常，请稍后再试" |

### 6.2 网络层全局拦截器逻辑

```
收到 API 响应
  │
  ├─ 状态码 == 423 && code == FORCE_CHANGE_PASSWORD
  │   └─ 拦截器处理：
  │       ├─ 不执行 refresh→retry 逻辑
  │       ├─ 取消当前请求的错误回调（避免业务层弹出 TDToast 错误提示）
  │       └─ 跳转至 [强制改密页]
  │
  ├─ 状态码 == 401（token 失效，AUTH_EXPIRED / AUTH_INVALID）
  │   └─ 拦截器处理：
  │       ├─ 尝试 POST /api/auth/refresh
  │       │   ├─ 成功 → 更新 accessToken → 重试原请求
  │       │   └─ 失败 → 清空 Token → 跳转 [登录页]
  │       └─ （注意：若当前已在改密页，不执行 refresh）
  │
  └─ 其他状态码 → 正常业务层处理
```

## 7. 边界情况与备注

### 7.1 边界情况

| 场景 | 处理方式 |
|------|---------|
| **新密码前后含空格** | 不自动 trim（密码中的空格是有效字符），但提示用户避免首尾空格 |
| **密码粘贴** | 允许粘贴，粘贴后密码默认不可见 |
| **快速连续点击确认** | 按钮 loading 态阻止重复提交 |
| **改密过程中 APP 切至后台** | 保持页面状态；若请求正在进行，回来后等待响应或超时 |
| **改密成功后 Token 立即失效** | 预期行为：后端自增 `tv` 使旧 Token 失效，改密成功后 APP 主动清空本地 Token |
| **改密页收到 401（token 失效，AUTH_EXPIRED）** | 说明 Token 已被 `tv` 机制失效，清空 Token 跳转登录页（而非 refresh） |
| **网络切换导致请求中断** | 提示网络异常，允许重试 |
| **深色模式** | 通过 `TDTheme.needMultiTheme()` 启用，所有颜色使用 TDesign Token，自动适配 |
| **键盘遮挡** | 使用 `resizeToAvoidBottomInset: true` + `SingleChildScrollView` 确保表单可见 |
| **无障碍** | 输入框设置 `semanticsLabel`；焦点顺序：新密码 → 确认密码 → 确认按钮 |
| **改密页被系统回收后恢复** | 从状态管理（如 `PageStorage`）恢复输入框内容（密码除外，安全考虑不恢复） |

### 7.2 完整认证流程（端到端）

```
管理员在后台重置用户密码
  → 后端设置 user.mustResetPassword = 1
  → 后端自增 tv（旧 Token 全部失效）

用户打开 APP / Token 过期
  → 跳转登录页

用户使用新密码登录
  → POST /api/auth/login → 成功 (200)
  → 响应：{ accessToken, refreshToken, user: { mustResetPassword: 1 } }
  → APP 检测 mustResetPassword == 1
  → 跳转 [强制改密页]（携带 accessToken）

用户在改密页设置新密码
  → POST /api/auth/change-password { newPassword }
  → 成功 (200)
  → 后端自增 tv → 当前 accessToken 失效
  → APP 清空本地 Token
  → TDToast "密码修改成功，请重新登录"
  → 跳转 [登录页]

用户使用新密码再次登录
  → POST /api/auth/login → 成功 (200)
  → 响应：{ accessToken, refreshToken, user: { mustResetPassword: 0 } }
  → mustResetPassword == 0 → 跳转 [首页]
  → 正常使用 APP
```

### 7.3 兜底防护机制

```
假设用户通过某种方式绕过了登录时的 mustResetPassword 检测
（例如：旧版本 APP 未实现该检测逻辑）

用户进入首页，发起任意业务 API 请求
  → 后端 authMiddleware 检测 mustResetPassword == 1
  → 返回 423 FORCE_CHANGE_PASSWORD
  → APP 网络层全局拦截器捕获
  → 不走 refresh→retry 路径（避免死循环）
  → 直接跳转 [强制改密页]

这确保了即使客户端遗漏了主路径检测，兜底机制也能生效。
```

### 7.4 安全要求

| 要求 | 说明 |
|------|------|
| **传输安全** | 所有请求通过 HTTPS 传输 |
| **密码不回显** | 改密成功后不清空输入框内容，但跳转页面后输入框内容不保留 |
| **Token 清除** | 改密成功后必须清除本地所有 Token 和用户缓存 |
| **防重放** | 改密请求不可重试（使用一次性 Token），失败后需重新登录 |
| **不存储密码** | 新密码仅在请求体中传输，不做任何本地持久化 |
| **改密页不可跳过** | 返回按钮需二次确认（TDDialog）；系统返回键同样拦截并弹出确认对话框 |

### 7.5 性能要求

| 指标 | 目标值 |
|------|--------|
| 页面启动到可交互 | < 300ms |
| 密码强度计算延迟 | < 16ms（每帧内完成） |
| 改密请求响应时间 | < 3s（正常网络） |
| 请求超时阈值 | 30s |
| TDToast 展示到跳转登录页 | 1.5s |

### 7.7 与登录页的关联

| 关联项 | 说明 |
|--------|------|
| **页面跳转** | 登录页 → 改密页（携带 accessToken）；改密页 → 登录页（清空 Token） |
| **Token 传递** | 改密页使用登录响应中的 accessToken 调用改密接口 |
| **状态隔离** | 改密页不依赖登录页的表单状态，两者完全独立 |
| **错误码共享** | `423` 状态码在登录页表示"账号锁定"（`ACCOUNT_LOCKED`），在改密接口表示"需强制改密"（`FORCE_CHANGE_PASSWORD`），需根据 `code` 字段区分 |
