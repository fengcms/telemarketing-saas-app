## 1. 页面概述

### 1.1 页面定位

线索列表页是电销工作台 APP 的核心页面之一，为不同角色（TE / TM / TA）提供线索数据的集中展示与检索能力。用户可在此页面浏览、搜索、筛选、排序线索，并点击进入线索详情。

### 1.2 目标用户

| 角色 | 数据范围 | 说明 |
|------|----------|------|
| TE（电销坐席） | `scope=mine` | 仅查看自己名下的线索 |
| TM（电销主管） | `scope=all` | 查看团队全部线索 |
| TA（电销管理员） | `scope=all` | 查看团队全部线索 |

### 1.3 核心目标

- 快速浏览线索列表，支持无限滚动分页
- 通过搜索、筛选、排序精准定位目标线索
- 通过卡片信息快速判断线索状态与跟进优先级
- 点击进入线索详情，开展跟进操作

### 1.4 全局排除规则

所有请求均自动附加以下参数，无需用户干预：

- `erased=0` — 排除已擦除记录
- 后端自动排除 `converted`（已转化）和 `is_blocked=1`（禁拨）的线索

## 4. 交互流程

### 4.1 首屏加载

```
用户操作                     系统响应
───────────────────────────────────────────────
1. 点击底部 Tab「线索」  →  2. 展示 TDNavBar + TDSkeleton 占位
                            3. 根据角色确定 scope 参数
                            4. 发起 GET /api/tenant/leads?page=1&size=20&erased=0
                            5. 接口返回 → 隐藏 TDSkeleton → 渲染线索卡片列表
                            6. 若总数 > 20 → 底部显示 TDLoadMore
                            7. 若总数 = 0 → 显示 TDEmpty 空态
```

### 4.2 搜索交互

```
用户操作                              系统响应
──────────────────────────────────────────────────────────
1. 点击搜索框                      → 2. TDSearchBar 获取焦点，弹出软键盘
                                        搜索框背景色变为 `gray-13`，shadowMedium 提升
3. 输入文字（每个字符）            → 4. 启动/重置 500ms 防抖计时器
5. 停止输入 ≥ 500ms               → 6. 自动发起搜索请求
                                       GET /api/tenant/leads?q={keyword}&erased=0
                                       (保留当前筛选/排序条件，叠加 q 参数)
                                    → 7. 显示 TDSkeleton 替换列表区域
                                    → 8. 接口返回 → 替换列表内容
                                    → 9. 若结果为 0 → 显示 TDEmpty 搜索空态
10. 按键盘回车                     → 11. 立即触发搜索（不等防抖）
                                       (行为同步骤 6)
12. 点击清除按钮 (X)              → 13. 清空搜索文本
                                    → 14. 恢复原始列表（保留筛选/排序条件）
                                    → 15. 搜索框恢复占位文字
```

### 4.3 筛选交互

```
用户操作                              系统响应
──────────────────────────────────────────────────────────
1. 点击漏斗图标                    → 2. 弹出筛选面板 TDPopup
                                        已选条件高亮显示（TDCheckTag 选中态）
3. 选择/取消筛选条件               → 4. 实时更新 TDPopup 内 TDCheckTag 选中态
5. 点击「确定」                    → 6. 关闭 TDPopup
                                    → 7. 在筛选标签栏生成对应 TDCheckTag
                                    → 8. 发起请求，参数使用通用查询 DSL
                                       如: status__in=following,assigned&categoryId=3
                                       叠加 erased=0 及当前搜索/排序条件
                                    → 9. 列表区域显示 TDSkeleton → 渲染新列表
10. 点击「重置」                   → 11. 清空 TDPopup 内所有 TDCheckTag 选中态
12. 点击 TDCheckTag 上的 X        → 13. 移除该筛选条件
```

## 6. 接口依赖

### 6.1 获取线索列表

```
GET /api/tenant/leads
```

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `scope` | string | 自动 | TE: `mine`；TM/TA: `all` |
| `page` | int | 否 | 页码，默认 1 |
| `size` | int | 否 | 每页条数，默认 20 |
| `erased` | int | 自动 | 固定 `0`，排除已擦除 |
| `q` | string | 否 | 搜索关键词（姓名/电话/公司） |
| `status__in` | string | 否 | 状态筛选，逗号分隔，如 `pending,following`（枚举见 §6.2） |
| `categoryId` | string | 否 | 分类筛选（裸字段即等值匹配；取值来自 `options/categories` 的 id） |
| `projectId` | string | 否 | 项目筛选（裸字段即等值匹配；取值来自 `options/projects` 的 id） |
| `updatedAt__gte` | int | 否 | 时间段筛选-起始（Unix 秒，见 [00-全局API约定](./00-全局API约定.md) §7） |
| `updatedAt__lte` | int | 否 | 时间段筛选-截止（Unix 秒） |
| `sort` | string | 否 | 排序字段（驼峰，`-` 前缀为降序），默认 `-updatedAt`；可选 `nextFollowupAt`（升序） |

**响应结构（成功 200）：**（统一信封 `{ success, data, error }` + 分页体 `{ items, total, page, size, pages }`，见 [00-全局API约定](./00-全局API约定.md) §1/§2）

```json
{
  "success": true,
  "data": {
    "items": [
      {
        "id": 10001,
        "name": "张*明",
        "phone": "138****1234",
        "company": "XX科技有限公司",
        "status": "following",
        "category": "住宅",
        "project": {
          "id": 5,
          "name": "阳光花园"
        },
        "owner": {
          "id": 3,
          "name": "李四"
        },
        "lastFollowupAt": 1752816600,
        "nextFollowupAt": 1752832800,
        "updatedAt": 1752816600
      }
    ],
    "total": 156,
    "page": 1,
    "size": 20,
    "pages": 8
  },
  "error": null
}
```

> 时间字段（`lastFollowupAt` / `nextFollowupAt` / `updatedAt`）为 **int Unix 秒**（见 [00-全局API约定](./00-全局API约定.md) §7），展示层格式化为相对时间或 `YYYY-MM-DD HH:mm`。

### 6.2 获取筛选选项（下拉选项接口）

筛选面板选项**不使用** `leads/filters`（该接口不存在），统一调用轻量下拉选项接口（见 [00-全局API约定](./00-全局API约定.md) §5 选项接口表）：

| 用途 | 接口 | 返回（`data` 内） |
|------|------|------|
| 分类选项 | `GET /api/tenant/options/categories` | `[{ id, name }]` |
| 项目选项 | `GET /api/tenant/options/projects` | `[{ id, name }]` |
| 归属人选项（仅 TM/TA 筛选面板） | `GET /api/tenant/options/users` | `[{ id, name, role }]` |

**响应结构（成功 200，信封 `{ success, data, error }`，`data` 直接为数组）：**

```json
{
  "success": true,
  "data": [
    { "id": 5, "name": "阳光花园" }
  ],
  "error": null
}
```

> **状态筛选为前端静态枚举**（不通过接口获取），取值与后端一致（见 [00-全局API约定](./00-全局API约定.md) §4.2）：
> `pending`（待跟进）/ `assigned`（已分配）/ `following`（跟进中）/ `converted`（已转化）/ `invalid`（无效）
>
> 选项数据需缓存至本地，避免每次打开筛选面板都请求。

## 7. 边界情况与备注

### 7.1 边界情况

| 场景 | 处理方式 |
|------|----------|
| 搜索关键词含特殊字符 | URL encode 后传递，前端不做截断 |
| 搜索关键词超长 | 搜索框内文字水平滚动，不截断 |
| 筛选条件超过 5 个 | TDCheckTag 区域横向滚动，不折行 |
| 卡片中项目名超长 | 单行省略号截断，tooltip 不展示（移动端不适用） |
| 快速连续上拉加载 | 加锁防止重复请求，前一次请求未完成时不发起新请求 |
| 搜索+筛选+排序叠加 | 所有条件叠加传递，参数互不干扰 |
| 清除搜索时列表为空 | 恢复原始列表（保留筛选/排序），若原始列表也为空则显示 TDEmpty |
| 切换 Tab 再返回 | 保留列表滚动位置与筛选/搜索状态 |
| 接口返回 401/403 | 跳转登录页或显示权限不足提示 |
| 接口返回 500 | 显示错误态，提供重试按钮 |
| 线索在列表中已被删除 | 点击进入详情时由详情页处理（显示已删除提示） |
| 夜间禁拨时段 | 列表页不做限制，由详情页/拨号时校验 |
| nextFollowupAt 恰好为今天 00:00 | 显示"今日可打" |
| TM/TA 查看列表 | 卡片底部多显示归属人信息行 |

### 7.2 备注

- 所有颜色值均引用 TDesign Flutter 设计令牌（Design Tokens），实际开发中通过 `TDTheme.of(context)` 获取，不硬编码 HEX 值。上文中 HEX 值仅供设计参考。
- 脱敏规则由后端返回，前端仅做展示，不自行脱敏。
- TDSkeleton 动画使用 Shimmer 效果，TDesign Flutter 内置支持。
- 无限滚动使用 `TDLoadMore` 组件，触发阈值：距离列表底部 5 个卡片高度时预加载。
- 筛选面板中的选项数据来源于下拉选项接口（`options/categories`、`options/projects`，TM/TA 另含 `options/users`，见 §6.2），需缓存至本地，避免每次打开面板都请求。
- 排序状态需要在切换 Tab 后保留。
- 搜索防抖仅在用户输入时触发；按回车键会立即触发搜索（取消待执行的防抖请求）。
