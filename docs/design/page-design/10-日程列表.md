## 1. 页面概述

| 项目 | 说明 |
|------|------|
| **页面定位** | 电销工作台 APP 底部 Tab 第 3 个入口「日程」，作为 TE/TM/TA 的待办日程管理中心 |
| **目标用户** | TE（电销员）、TM（电销主管）、TA（电销管理员） |
| **核心目标** | 1. 快速查看今日及未来的待办日程，掌握工作节奏；2. 按状态筛选待办/已完成日程；3. TM/TA 可切换查看团队全员日程；4. 列表内"已逾期"分组标红置顶，逾期情况一目了然（**到期提醒的主入口在首页提醒条**，见 `03-首页看板.md`，本页不做独立的弹窗/横幅提醒） |
| **入口** | 底部导航栏 Tab 3「日程」，Tab 图标右上角显示 Badge（今日待办数量，来自 `GET /api/tenant/schedules/stats/mine` 的 `byStatus.dueToday`） |
| **数据权限** | TE：仅自己的日程（请求时传 `userId={当前用户ID}`，见 §3.5）；TM/TA：可查看"我的"或"团队全部" |

## 3. 组件规格

### 3.1 TDNavBar（顶部导航栏）

| 属性 | 规格 |
|------|------|
| 组件 | `TDNavBar` |
| 高度 | 56dp |
| 背景色 | `brand-7`（#0052D9） |
| 标题文字 | "日程"，`fontTitleMedium`（18sp），SemiBold，`gray-13`（#FFFFFF） |
| 左侧图标 | 无（或汉堡菜单，取决于全局导航设计） |
| 右侧图标 | 🔔 通知入口（可选）；**无搜索图标**（日程列表 MVP 不做搜索） |
| 阴影 | 无（`shadowElevation: 0`，使用 `shadowBase` 微弱分层） |

### 3.2 到期提醒（主入口在首页，本页不设提醒条）

> **产品决策**：日程到期提醒的主入口是**首页提醒条**（前台每 10 分钟轮询、30 分钟内到期提示，见 `03-首页看板.md`）。日程列表页**不做**独立的弹窗/横幅/通知条提醒，仅在「待办」Tab 内以红色"已逾期"分组头置顶展示逾期日程（见 §3.6.1），卡片样式见 §3.7.2。

### 3.3 搜索（MVP 不实现）

> **产品决策**：日程列表 MVP 不做搜索，本页不设搜索框。后端 `GET /api/tenant/schedules` 已支持 `q` 参数（搜索 `schedules.title` + `schedules.content` + `leads.name`），后续版本启用搜索时，输入防抖统一为 **500ms**（与线索列表一致，见评审 SYS-06 处置）。

### 3.4 TDTabBar（筛选 Tab — 状态筛选）

| 属性 | 规格 |
|------|------|
| 组件 | `TDTabBar`（固定 **2 个 Tab**） |
| 高度 | 48dp |
| Tab 项 | 「待办」「已完成」（**"已逾期"不是独立 Tab**，在「待办」Tab 内以红色分组头置顶展示，见 §3.6.1） |
| Tab 角标 | 「待办」角标 = `byStatus.pending`；「已完成」角标 = `byStatus.completed`（来自 `GET /api/tenant/schedules/stats/mine`，见 §6.2）；角标为 0 时隐藏；超过 99 显示 "99+" |
| 选中态 | 文字 `fontBodyMedium`（14sp），Medium，`brand-7`（#0052D9）；底部指示条 2dp，`brand-7` |
| 未选中态 | 文字 `fontBodyMedium`（14sp），Regular，`gray-6`（#A6A6A6） |
| 指示条动画 | 滑动过渡，200ms，`Curves.easeInOut` |
| 对应 API 参数 | 待办 → `status=pending`；已完成 → `status__in=completed,cancelled`（已完成/已取消均为终态，卡片以状态标签区分） |

## 4. 交互流程

### 4.1 页面初始化

```
用户操作                          系统响应
─────────                        ─────────
1. 点击底部 Tab「日程」     →    2. 页面进入，显示 TDSkeleton 骨架屏
                            →    3. 并行请求：
                                   a. GET /api/tenant/schedules?status=pending&sort=scheduledAt&page=1&size=20
                                   b. GET /api/tenant/schedules/stats/mine
                                      （byStatus 用于 Tab 角标与底部 Badge，见 §6.2）
                            →    4. 数据返回后，按 scheduledAt 日期分组渲染卡片
                                 （逾期条目归入"已逾期"红色分组置顶，见 §3.6.1）
                            →    5. TDSkeleton 淡出（200ms），列表淡入
```

> 到期提醒轮询不在本页：前台每 10 分钟轮询并在首页顶部展示提醒条，见 `03-首页看板.md`。

### 4.2 筛选 Tab 切换

```
用户操作                          系统响应
─────────                        ─────────
1. 点击「已完成」Tab        →    2. TDTabBar 指示条滑动到「已完成」（200ms 动画）
                            →    3. 显示 TDSkeleton 骨架屏（局部，仅列表区域）
                            →    4. 请求 GET /api/tenant/schedules?status__in=completed,cancelled&sort=-scheduledAt&page=1&size=20
                            →    5. 数据返回后重新分组渲染（已完成 Tab 无"已逾期"分组）
                            →    6. 列表滚动到顶部
```

### 4.3 范围切换（TM/TA）

```
用户操作                          系统响应
─────────                        ─────────
1. 点击「团队日程」TDCheckTag →  2. TDCheckTag 切换动画
                            →    3. 请求 GET /api/tenant/schedules?status=pending&sort=scheduledAt&page=1&size=20
                                 （不传 userId，TM/TA 后端返回团队全部日程）
                            →    4. 卡片中显示"归属：XXX"字段
                            →    5. 列表刷新，滚动到顶部
```

### 4.4 搜索（MVP 不实现）

> 日程列表 MVP 不做搜索，无搜索交互流程。后端 `q` 参数能力保留，后续版本启用时见 §3.3 说明。

### 4.5 点击日程条目

```
用户操作                          系统响应
─────────                        ─────────
1. 点击某日程卡片           →    2. 卡片产生点击反馈（TDesign 内置水波纹）
                            →    3. 跳转到日程详情页，传入 scheduleId
```

### 4.6 下拉刷新

```
用户操作                          系统响应
─────────                        ─────────
1. 列表顶部下拉            →    2. 显示 TDRefreshHeader 刷新指示器
                            →    3. 请求 GET /api/tenant/schedules?{当前筛选参数}&page=1&size=20
                                 并重新请求 GET /api/tenant/schedules/stats/mine
                            →    4. 数据返回后替换列表数据，TDRefreshHeader 消失
                            →    5. 同步更新 Tab 角标与底部 Badge 数字
```

### 4.7 到期提醒（本页不实现，见首页）

## 6. 接口依赖

### 6.1 获取日程列表（主接口）

> 响应信封、分页、过滤/排序参数命名以 [00-全局API约定](./00-全局API约定.md) 为准。

```
GET /api/tenant/schedules
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `status` | string | 否 | 状态等值过滤：`pending` / `completed` / `cancelled`；不传时后端默认 `pending` |
| `status__in` | string | 否 | 状态多选，逗号分隔：如 `completed,cancelled`（与 `status` 二选一） |
| `sort` | string | 否 | 排序字段（驼峰，`-` 前缀为降序），默认 `scheduledAt`（升序） |
| `userId` | string | 否 | 归属人 ID；TE 由后端强制过滤为本人（传否无效）；TM/TA "我的日程"传自己 ID，"团队日程"不传 |
| `scheduledAt__lte` | int | 否 | 计划时间上界（**Unix 秒**），用于逾期/到期筛选 |
| `scheduledAt__gte` | int | 否 | 计划时间下界（**Unix 秒**） |
| `page` | int | 否 | 页码，从 1 起，默认 1 |
| `size` | int | 否 | 每页条数，默认 20，最大 200 |

> MVP 不启用搜索（不传 `q`）；后端保留 `q` 参数能力（搜索 title/content/线索姓名）。

**响应结构（对齐 00 信封）：**

```json
{
  "success": true,
  "data": {
    "items": [
      {
        "id": "sch_001",

        "userId": "user_001",
        "leadId": "lead_001",
        "callRecordId": null,
        "title": "跟进张先生购房意向",
        "content": "客户对三居室感兴趣",
        "status": "pending",
        "scheduledAt": 1784008800,
        "completedAt": null,
        "createdAt": 1783908000,
        "updatedAt": 1783908000,

          "name": "张伟",
          "phone": "13812341234"
        }
      }
    ],
    "total": 56,
    "page": 1,
    "size": 20,
    "pages": 3
  },

```

> 字段说明（对齐后端 `src/routes/tenant/schedules.ts`）：
> - 时间字段（`scheduledAt`/`completedAt`/`createdAt`/`updatedAt`）为 **int Unix 秒**，展示层格式化为 `YYYY-MM-DD HH:mm`（见 00 §7）
> - 列表仅返回 `userId`（归属人 ID），**不返回归属人姓名对象**；"归属：XXX" 展示由前端用 `GET /api/tenant/options/users` 缓存做本地映射
> - `lead` 仅含 `{name, phone}`（线索已擦除时为 `null`）；电话为明文（REQ-08 已上线，日程列表不脱敏）

### 6.2 我的日程统计（Tab 角标 / 底部 Badge / 逾期分组计数）

```
GET /api/tenant/schedules/stats/mine
Authorization: Bearer {token}

成功响应 (200)：
{
  "success": true,
  "data": {
    "byStatus": {
      "pending": 8,        // 「待办」Tab 角标
      "completed": 12,     // 「已完成」Tab 角标
      "cancelled": 3,
      "overdue": 2,        // "已逾期"分组头计数（pending 且 scheduledAt ≤ now）
      "dueToday": 5        // 底部 TabBar Badge（今日待办数，pending 且 scheduledAt ≤ 今日 23:59:59 北京时间）
    }
  },
  "error": null
}
```

| 字段 | 用途 |
|------|------|
| `byStatus.pending` | 「待办」Tab 角标数字 |
| `byStatus.completed` | 「已完成」Tab 角标数字 |
| `byStatus.overdue` | 「待办」Tab 内"已逾期"分组头的 N（§3.6.1） |
| `byStatus.dueToday` | 底部导航栏日程 Tab Badge（§3.8） |

> 本接口替代旧版"轮询取 total 做提醒"的方案：到期提醒改由首页统一轮询（见 03），本页仅在初始化与下拉刷新时请求本接口刷新角标。

## 7. 边界情况与备注

### 7.1 边界情况

| 场景 | 处理方式 |
|------|----------|
| 日程无关联线索（`lead` 为 null） | 卡片中线索区域显示"无关联线索"，`gray-6` 灰色文字 |
| 日程标题为空 | 显示默认标题"未命名日程" |
| 同一天内日程超过 20 条 | 日期分组正常显示，分页按总列表分页而非按分组 |
| scheduledAt 恰好在今天 23:59 | 归入"今天"分组 |
| scheduledAt ≤ 当前时间且 status=pending | 归入「待办」Tab 顶部"已逾期"红色分组（§3.6.1），卡片显示"已逾期" |
| 网络断开后恢复 | 自动触发一次下拉刷新 |
| 快速连续切换筛选 Tab | 取消前一个请求（dio `CancelToken`），仅保留最后一次 |
| Badge 数字与实际列表不一致 | 以 stats/mine 返回为准，列表加载完成后同步刷新角标 |
| 跨天使用（用户长时间不关闭 APP） | 每小时重新计算日期分组与"已逾期"分组归属，必要时刷新列表 |
| TM/TA 切换到团队日程后切回"我的" | 重置 page=1，重新请求 `userId={当前用户ID}` |

### 7.2 备注

| 项目 | 说明 |
|------|------|
| 日期分组算法 | 今天：`scheduledAt` 日期 = 当前日期；明天：当前日期 + 1；后天：当前日期 + 2；本周：当前日期 + 3 ~ 本周日；更早：超出本周日的 |
| 排序规则 | 「待办」Tab："已逾期"分组整体置顶（组内按 `scheduledAt` 升序，逾期最久排最前），其余日期分组按时间先后排列，同组内按 `scheduledAt` 升序 |
| 性能优化 | 列表使用 `ListView.builder` + `AutomaticKeepAliveClientMixin`，配合 `AnimatedList` / `SliverAnimatedList` 做增量更新，避免全量刷新 |
| 无障碍 | 卡片支持 TalkBack 朗读：标题 + 线索姓名 + 时间 + 状态 |
| 暗色模式 | 所有颜色使用 TDesign Token，通过 `TDTheme.needMultiTheme()` 自动适配暗色模式 |
