## 1. 页面概述

### 1.1 页面定位

通话结果反馈面板是 APP 最核心的交互枢纽。用户在系统拨号盘完成通话后切回 APP，面板自动弹出，承载「通话记录自动回传 + 跟进信息手动补充」的完整流程。面板以 `TDPopup` 形式呈现，不离开当前线索上下文。

### 1.2 目标用户

| 角色 | 使用场景 |
|------|---------|
| **TE（租户员工）** | 每次拨号后必用，日均 30-80 次 |
| **TM（租户经理）** | 偶尔拨号后使用 |
| **TA（租户管理员）** | 偶尔拨号后使用 |

### 1.3 核心目标

- **减少手动输入**：自动读取通话记录，填充时长/时间，用户只需选类型 + 写备注
- **流程闭环**：一次面板操作完成「通话记录 + 跟进备注 + 下次日程」三步
- **数据不丢失**：draft 机制确保 APP 被杀、网络断开等异常情况下数据可恢复

### 2.5 按接听类型的渐进式披露

> 面板根据所选「接听类型」智能调整可见区域：失败通话下隐藏低价值字段，让员工聚焦于真正有意义的动作。
> 核心原则：**没接通就没有对话，跟进备注冗余；号码失效（空号/停机）则回拨无意义，日程也隐藏；线索分类始终保留**（失败通话下标注"无价值客户"恰恰是最有价值的动作）。

**三档状态规则：**

| 接听类型 | 跟进备注 + 快捷备注 | 新建日程按钮 | 线索分类 | 提交按钮文字 |
|---------|---------------------|--------------|---------|--------------|
| 已接听 | 展开（完整） | 显示 | 可操作 | 提交反馈 |
| 无人接听 / 拒接 | 折叠（留展开入口） | 显示（可选回拨） | 可操作 | 记录并继续 |
| 空号 / 停机 | 折叠（留展开入口） | 隐藏 | 可操作（引导标无效） | 记录并继续 |

**状态 A：已接听（完整态）** —— 即 §2.1 主布局，所有区域展开。

**折叠/展开交互：**

| 行为 | 说明 |
|------|------|
| 默认折叠 | 接听类型为失败类时，跟进备注 TDTextarea 与快捷备注区域默认收起 |
| 展开入口 | 显示一行文字按钮 "＋ 补充备注"，`fontBodyMedium`（14sp），`brand-7` 色 |
| 点击展开 | 展开 TDTextarea + 快捷备注区域（200ms 高度动画），入口文字变为 "－ 收起备注" |
| 再次点击 | 收起；若已输入内容，收起前不清空（draft 保留） |
| 切回已接听 | 备注区域自动恢复展开态 |
| 日程隐藏 | 空号/停机时新建日程按钮不渲染；如需建日程可在线索详情页独立入口创建 |
| 分类引导语 | 空号/停机时分类标签后追加提示 "（建议标注为无效客户）"，`fontBodyExtraSmall`（12sp），`gray-6` |

## 4. 交互流程

### 4.1 主流程：正常通话后反馈

```
用户操作                          系统响应
─────────                        ─────────
1. 在线索详情页点击 [拨号]         
                                  → 生成 externalCallId (UUID v4)
                                  → 写入本地 draft：
                                    {leadId, phone, externalCallId,
                                     startedAt: now, draft: true}
                                  → 调起系统拨号盘 (Intent.ACTION_DIAL，预填号码)
                                  
2. 通话结束，用户切回 APP          
                                  → APP 从后台恢复 (onResume)
                                  → 检测 READ_CALL_LOG 权限
                                  → 读取最近通话记录 (ContentResolver)
                                  → 匹配条件：
                                    a) 号码一致（最后 11 位匹配）
                                    b) 通话开始时间在 2 分钟内
                                    c) externalCallId 与 draft 一致
                                  → 匹配成功 ✓
                                  → 更新 draft：
                                    {endedAt, duration, answerType(推断)}
                                  → 弹出 TDPopup 反馈面板
                                  → 自动填充：
                                    - 紧凑头部行：号码（脱敏）+ 通话时长
                                    - startedAt / endedAt
                                  → 接听类型：根据 duration 预选
                                    (duration > 0 → "已接听"，
                                     duration = 0 → "无人接听")
                                  → 线索分类：默认选中线索当前分类

3. 用户确认/修改接听类型           
   点击 TDCheckTag 选择             
                                  → 选中态切换（动画 200ms）+ 轻震动
                                  → 必选校验：至少选一个
                                  → 按接听类型渐进式披露（详见 §2.5）：
                                    ├─ 已接听：备注/快捷备注展开，日程按钮显示，
                                    │         按钮文字「提交反馈」
                                    ├─ 无人接听/拒接：备注折叠（留"＋补充备注"入口），
                                    │         日程按钮显示（可选回拨），按钮「记录并继续」
                                    └─ 空号/停机：备注折叠，日程按钮隐藏，
                                              分类标签追加"（建议标注为无效客户）"，
                                              按钮「记录并继续」
                                  → 切换回已接听时，备注区域恢复展开态

4. 用户修改线索分类（可选）         
   点击平铺的分类 TDCheckTag        
                                  → 切换选中分类（轻震动）
                                  → 再次点击已选中 Tag → 取消选中（保持原分类）

5. 用户输入跟进备注（可选）         
   在 TDTextarea 中输入文字        
                                  → 实时显示字数统计
                                  → 达到 2000 字时禁止继续输入

   [或] 点击平铺的快捷备注 Tag      
                                  → Tag 高亮 200ms 反馈 + 轻震动
                                  → 内容追加到备注文本域末尾
                                  → 可连续点击多个 Tag 组合内容

6. 用户新建跟进日程（可选）         
   点击 [＋ 新建跟进日程] 按钮      
                                  → 打开「12-新建日程」页面
                                    传参：leadId + 当前备注内容（预填 content）
   在 12 页面设置时间/标题 → 创建   
                                  → POST /api/tenant/schedules（由 12 页面发起）
                                  → 创建成功返回面板
                                  → 面板显示回显行：
                                    "已预约：{MM-DD HH:mm} {标题/内容摘要}"
   （不需要日程则跳过此步）         

7. 用户点击 [提交反馈]/[记录并继续]  
                                  → 校验：接听类型已选？
                                    → 未选：TDCheckTag 区域抖动 + TDToast "请选择接听类型"
                                    → 已选：继续
                                  → 提交按钮进入 loading 态
                                  → 发起 API 调用（见 §6）：
                                    POST /api/tenant/calls
                                    （携带 answerType/duration/categoryId/content，
                                     后端原子创建通话记录 + 跟进记录）
                                  → 注：日程已在步骤 6 由 12 页面独立创建，
                                    提交不再夹带日程创建
                                  
                                  → 成功：
                                    a) 清除本地 draft
                                    b) TDToast "提交成功"
                                    c) TDPopup 下滑动画关闭 (300ms)
                                    d) 刷新线索详情页时间线

8. 面板关闭                        
                                  → 回到线索详情页
                                  → 时间线顶部显示新提交的跟进记录
                                  → 若存在 listContext（从列表页进入）：
                                    详情页底部显示「✓ 已提交  [查看下一个 →]」
                                    临时提示条（3 秒），点击跳转下一条线索
                                    （详见 05-线索详情 §4.10）
```

### 4.4 提交失败处理

> 日程已在「12-新建日程」页面独立创建，面板提交仅为单接口
> `POST /api/tenant/calls`（后端原子创建通话记录 + 跟进记录），
> 因此不存在跨接口的部分失败，处理逻辑大幅简化。

```
用户点击 [提交反馈]/[记录并继续]
                                  → POST /api/tenant/calls
                                  → 失败（网络错误）：
                                    自动重试 1 次（间隔 1s）
                                  → 重试仍失败：
                                    提交按钮恢复可点击
                                    TDToast "提交失败，请检查网络后重试"
                                    面板保持打开，已填数据不丢失（draft 保留）
                                  → 接口返回 409（externalCallId 幂等冲突）：
                                    视为成功（说明上次实际已提交）
                                    正常关闭面板
                                  → 接口返回 404（线索已删除/转移）：
                                    TDDialog "该线索已被删除或转移"
                                    → [确定] → 清除 draft → 关闭面板 → 返回列表页

注：若用户在步骤 6 创建了日程、但步骤 7 提交通话记录失败，
    日程仍然有效（独立的跟进意图），不受通话记录提交结果影响。
```

## 6. 接口依赖

### 6.1 提交通话结果（主接口）

> 面板提交的核心接口。后端在创建通话记录的同时，根据 `content` 原子创建对应的跟进记录，
> 前端无需再单独调用跟进接口。日程不在此接口创建（由「12-新建日程」页面独立处理）。

```
POST /api/tenant/calls
Content-Type: application/json
Authorization: Bearer {token}

请求体：
{
  "leadId": "uuid",
  "phone": "13812345678",
  "externalCallId": "uuid-v4",        // 幂等键
  "startedAt": 1753252200,             // int Unix 秒（展示层才用 YYYY-MM-DD HH:mm）
  "endedAt": 1753252353,               // int Unix 秒
  "duration": 153,                     // 秒
  "answerType": "answered",            // answered | no_answer | rejected | empty_number | suspended
  "categoryId": "uuid",               // 可选，修改后的线索分类（不传或传原值表示不变）
  "content": "客户表示感兴趣..."       // 可选，跟进备注（后端据此自动创建跟进记录）
}

成功响应 (201)：
{
  "id": "uuid",
  "created": true
}

幂等说明：
- 同一 externalCallId 重复提交返回 200 + 已有记录
- 响应中 "created": false 表示已存在
```

### 6.2 记录跟进备注（通常无需单独调用）

> 跟进记录由 `POST /api/tenant/calls` 根据 `content` 字段自动创建，面板提交流程中**不再单独调用**此接口。
> 仅在后端实现不支持自动创建跟进的特殊情况下，才作为兜底单独调用。

```
POST /api/tenant/leads/:leadId/followups
```

## 7. 边界情况与备注

### 7.1 权限相关

| 场景 | 处理 |
|------|------|
| READ_CALL_LOG 未授权 | 面板弹出，顶部提示条引导授权；用户可手动填写所有字段 |
| 用户选择"不再询问"拒绝 | 每次拨号后都提示手动填写；在设置页提供永久开启引导 |
| 通话记录读取超时（> 3s） | 放弃自动填充，直接让用户手动填写 |

### 7.2 并发与竞态

| 场景 | 处理 |
|------|------|
| 用户快速连续点击提交 | 第一次点击后按钮立即禁用 + loading，防止重复提交 |
| 面板打开期间线索被其他 TE 修改 | 提交时后端校验，返回 409 → 提示"线索信息已变更，请刷新后重试" |
| 多个 draft 同时存在 | 每个 leadId 最多一个 draft；新拨号覆盖旧 draft |

### 7.3 数据边界

| 场景 | 处理 |
|------|------|
| 通话时长 = 0 | 接听类型预选"无人接听"；时长显示"0秒"；提交按钮文字为"记录并继续" |
| 通话时长 > 4 小时 | 显示实际时长，但弹出 TDDialog 确认"通话时长异常（X小时），确认提交？" |
| 备注内容含特殊字符 | 前端不限制，后端做 XSS 过滤 |
| 备注内容为纯空格 | trim 后为空 → 视为未填写 |
| 线索当前分类不在选项列表中 | 分类 Tag 组不预选任何项（视为未分类），员工可手动选择 |
| 未接通快速提交 | 接听类型为失败类时，仅需接听类型即可提交；备注/分类均为可选，不强制 |
| 失败通话备注折叠 | 无人接听/拒接/空号/停机时备注与快捷备注默认折叠，显示"＋ 补充备注"入口 |
| 折叠态展开后已输入内容 | 点击"－ 收起备注"收起时不清空已输入内容（draft 保留），再次展开仍可见 |
| 空号/停机隐藏日程 | 新建日程按钮不渲染；员工如仍需建日程，可在线索详情页独立入口创建 |
| 空号/停机分类引导 | 分类标签后追加"（建议标注为无效客户）"提示语，引导标注无效 |
| 切换接听类型回已接听 | 备注区域自动恢复展开态；日程按钮恢复显示 |
| 自动预选触发的披露 | 面板打开时若按时长预选"无人接听"，立即进入对应折叠态，无需员工手动操作 |
| 新建日程后返回面板 | 12 页面关闭返回后，面板已填数据由 draft 保留，不丢失；显示「已预约」回显行 |
| 创建了日程但通话记录提交失败 | 日程独立有效（已写入后端），不随通话记录提交结果回滚；回显行保留 |
| 取消「已预约」回显行 | 点击 ✕ 确认后调用 DELETE /api/tenant/schedules/:id 删除日程，移除回显行 |
| 备注为空时点新建日程 | 12 页面 content 预填为空，员工可在 12 页面自行填写日程内容 |

### 7.4 无障碍

| 项目 | 要求 |
|------|------|
| TDCheckTag 选中 | 除颜色变化外，显示 ✓ 图标 + semanticsLabel "已选中" |
| TDTextarea | 必须有 semanticsLabel |
| 按钮 | 提交按钮 loading 时 semanticsLabel = "正在提交" |
| 触感反馈 | 所有可点选 Tag（接听类型/分类/快捷备注）及新建日程按钮点击时触发 `HapticFeedback.lightImpact()`，提供物理确认感 |
| 对比度 | 所有文字与背景对比度 ≥ 4.5:1（WCAG AA） |

### 7.5 性能

| 项目 | 要求 |
|------|------|
| 面板弹出延迟 | 从 onResume 到面板可见 < 300ms |
| 通话记录读取 | 超时 3s，超时后降级为手动填写 |
| 分类列表加载 | 面板首次打开时缓存，后续复用 |
| 快捷备注加载 | APP 启动时预缓存，面板打开时直接使用 |
