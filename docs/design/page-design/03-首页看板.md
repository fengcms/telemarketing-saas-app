## 1. 页面概述

### 1.1 页面定位

首页看板是 APP 的默认首屏页面（底部 Tab 1），用户打开 APP 后的第一个触点。页面聚合展示坐席当日核心工作指标与待办日程预览，帮助坐席快速了解"今天做了什么、还要做什么"。

### 1.2 目标用户

| 角色 | 使用场景 |
|------|---------|
| **TE（坐席员工）** | 每日开工查看今日数据概况和待办日程清单，规划当天拨打任务 |
| **TM（经理）** | 查看个人数据 + 通过右上角入口查看团队看板 |
| **TA（管理员）** | 同 TM，额外拥有团队看板入口权限 |

### 1.3 核心目标

- **信息聚合**：一屏内展示今日 4 项核心工作指标，无需跳转
- **待办驱动**：预览最近 5 条待办日程，引导坐席快速进入工作
- **快捷导航**：提供线索列表、通话记录两个高频入口，减少导航层级
- **数据时效**：通过 10 分钟轮询 + 下拉刷新保证数据新鲜度

#### 3.2.2 日程到期提醒条（ScheduleDueSoonBanner）

> 对应主文档 `APP_PRODUCT_DESIGN.md` §4.3.2「日程到期提醒」：APP 处于前台时每 **10 分钟**轮询一次，检测未来 **30 分钟内**到期的未完成日程（`status=pending` 且 `now ≤ scheduledAt ≤ now+30min`），在首页顶部展示非侵入式提醒条。

| 属性 | 规格 |
|------|------|
| **类型** | `TDNoticeBar`（自定义内容，可关闭） |
| **组件位置** | 首页顶部，TDNavBar（及离线提示条，如有）之下、可滚动区域之上；固定不随内容滚动 |
| **显示条件** | 轮询检测到 N ≥ 1 条「未来 30 分钟内到期的未完成日程」时显示；N = 0 时不显示 |
| **高度** | 36px（含上下 `paddingTop=spacer8(8px), paddingBottom=spacer8(8px)`） |
| **背景色** | `brand-1 #E0EAFF`（TDesign 品牌色浅底，非侵入式弱提示） |
| **图标** | `TDIcons.time`，`20px × 20px`，`color=brand-7 #0052D9`，左侧 `startMargin=spacer16(16px)` |
| **文字** | "您有 N 条日程即将到期"（N 为检测到条数），`fontBodySmall`（13sp），`textColor=brand-7 #0052D9`，`startMargin=spacer8(8px)`（距图标） |
| **关闭按钮** | 右侧 `TDIcons.close`，`16px × 16px`，`color=brand-7 #0052D9`，点击区域 `32px × 32px`，`endMargin=spacer8(8px)` |
| **点击行为** | 点击提醒条主体 → 切换底部 TDBottomTabBar 至「日程」（Tab 3），日程列表自动过滤 `status=pending` 且 `scheduledAt ≤ now+30min` |
| **动画** | 显示/隐藏从顶部滑入/滑出，`duration=200ms`，`Curves.easeOut`（与离线提示条一致） |

**消失条件（满足任一即隐藏）：**

| # | 条件 | 说明 |
|---|------|------|
| 1 | 全部处理完 | 轮询检测到 N = 0（目标日程均已 `completed`/`cancelled`） |
| 2 | 全部过期 | 目标日程 `scheduledAt` 均已早于当前时间（已转为「已逾期」，不再属于「即将到期」） |
| 3 | 用户关闭当次 | 用户点击 [✕] 后，**当次会话内**（APP 本次前台运行期间）不再重复提示；APP 重启或次日重新检测 |

**轮询机制：**

- 复用首页前台 10 分钟自动轮询（见 §4.7），不额外增加定时器频率。
- 检测请求：`GET /api/tenant/schedules?status=pending&scheduledAt__gte={now}&scheduledAt__lte={now+1800}&page=1&size=1`（`now` 为服务端时间 Unix 秒，见 §6.5），取响应 `data.total` 作为 N。
- 进入首页（首屏加载）与下拉刷新时也同步执行一次检测。
- 用户关闭提醒条后，仅当下一轮轮询检测到**新的**即将到期日程（ID 集合与上次不同）时才重新出现。

## 4. 交互流程

### 4.1 页面初始化加载

```
用户操作                          系统响应
───────────────────────────────────────────────────────
1. 点击底部 TDBottomTabBar「首页」→  2. 显示 TDSkeleton 骨架屏
                                3. 并行发起请求：
                                   a. GET /api/tenant/stats/mine?dateFrom=YYYY-MM-DD&dateTo=YYYY-MM-DD
                                   b. GET /api/tenant/schedules?status=pending&page=1&size=5&sort=scheduledAt
                                   c. GET /api/tenant/schedules/stats/mine
                                      （四宫格"今日待办" + 待办日程 Badge 数据源）
                                   d. GET /api/tenant/schedules?status=pending
                                      &scheduledAt__gte={now}&scheduledAt__lte={now+1800}
                                      &page=1&size=1
                                      （日程到期提醒条检测，见 §3.2.2）
                                4. 等待接口返回...
                                   │
                     ┌────────────┼────────────┐
                     ↓            ↓            ↓
                 [stats 先返回]  [sched 先返回] [同时返回]
                     │            │            │
                     ↓            ↓            ↓
               5a. 统计区域     5b. 日程区域   5c. 全部区域

```

## 5. 状态设计

### 5.1 加载态（Loading）

**首屏加载**：

| 区域 | 表现 |
|------|------|
| 统计卡片 | 4 个 TDSkeleton 占位，内置闪烁动画 |
| 日程区域 | 3 个 TDSkeleton 占位，内置闪烁动画 |
| 快捷入口 | 始终显示（不依赖接口，仅"我的线索"计数后加载） |
| 下拉刷新 | 不触发（首屏使用 TDSkeleton 骨架屏而非下拉动画） |

**局部加载**（某接口慢于另一接口）：

| 场景 | 表现 |
|------|------|
| stats 先返回，schedules 还在加载 | 统计区域显示数据，日程区域保持 TDSkeleton |
| schedules 先返回，stats 还在加载 | 日程区域显示数据，统计区域保持 TDSkeleton |

### 5.2 空态（Empty）

| 区域 | 触发条件 | 表现 |
|------|---------|------|
| 日程区域 | 接口返回 `data.items.length === 0` | 显示 TDEmpty + "暂无待办日程" + 副文案 |
| 快捷入口-我的线索 | `myLeadsTotal === 0` | 副标题显示 "0 条"，入口仍可点击 |

### 5.3 错误态（Error）

**首屏加载失败**：

| 区域 | 表现 |
|------|------|
| 统计区域 | 显示错误占位：`⚠ 加载失败` + `[重试]` TDButton(text) |
| 日程区域 | 显示错误占位：`⚠ 加载失败` + `[重试]` TDButton(text) |
| 快捷入口 | 不受影响（无接口依赖） |

**下拉刷新失败**：

| 场景 | 表现 |
|------|------|
| 两个接口均失败 | 显示 TDToast："数据刷新失败，请检查网络"，含重试机会 |
| 仅一个接口失败 | 成功的区域正常更新，失败的区域保持旧数据，TDToast 提示 "部分数据更新失败" |

**重试按钮规格**：

| 属性 | 规格 |
|------|------|
| **类型** | `TDButton(theme: TDButtonTheme.text)` |
| **文字** | "重试" |
| **字号** | `fontBodyMedium`（14sp） |
| **颜色** | `brand-7 #0052D9` |
| **点击行为** | 仅重新请求该区域对应的接口 |

### 5.4 离线态（Offline）

| 属性 | 规格 |
|------|------|
| **触发条件** | 设备网络连接断开（`connectivity_plus` 监听） |
| **表现** | TDNavBar 下方滑入 TDNoticeBar 离线提示条（见 3.2） |
| **数据展示** | 显示最后一次缓存的数据（`shared_preferences` 轻量缓存；本项目无本地数据库设计，不使用 sqflite） |
| **交互限制** | 下拉刷新禁用；点击日程跳转仍可（本地数据）；拨号功能不受影响（系统拨号盘） |
| **恢复网络** | 提示条滑出消失 → 自动触发一次数据刷新 → 刷新完成后恢复正常交互 |

### 5.5 Token 过期态

| 场景 | 表现 |
|------|------|
| accessToken 过期 | 网络层自动使用 refreshToken 换取新 token → 自动重试原请求 → 用户无感知 |
| refreshToken 也过期 | 跳转登录页，TDToast 提示 "登录已过期，请重新登录" |
| 423 FORCE_CHANGE_PASSWORD | 跳转强制改密页（详见 `APP_PRODUCT_DESIGN.md` 4.1.2） |

```
              ↓            ↓            ↓
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ 加载成功  │ │ 部分成功  │ │ 全部失败  │
       └────┬─────┘ └────┬─────┘ └────┬─────┘
            ↓            ↓            ↓
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ 正常展示  │ │ 数据+错误 │ │ 错误+重试 │
       │          │ │ 混合展示  │ │          │
       └────┬─────┘ └────┬─────┘ └────┬─────┘
            │            │            │
            ↓            ↓            ↓
       ┌──────────────────────────────────┐
       │          用户可操作状态            │
       │  · 下拉刷新 → 回到「骨架屏加载」  │
       │  · 10 分钟轮询 → 静默更新数据     │
       │  · 网络断开 → 离线态              │
       └──────────────────────────────────┘
```

## 6. 接口依赖

### 6.1 接口清单

| # | 接口 | 方法 | 用途 | 触发时机 |
|---|------|------|------|---------|
| API-01 | `/api/tenant/stats/mine` | GET | 获取今日个人统计（今日跟进/今日接通/我的线索） | 首屏加载、下拉刷新、10 分钟轮询 |
| API-02 | `/api/tenant/schedules` | GET | 获取待办日程预览 | 首屏加载、下拉刷新、10 分钟轮询 |
| API-03 | `/api/tenant/schedules/stats/mine` | GET | 我的日程统计（四宫格"今日待办" + 待办日程 Badge） | 首屏加载、下拉刷新、10 分钟轮询 |
| API-04 | `/api/tenant/schedules` | GET | 即将到期日程检测（日程到期提醒条，见 §3.2.2） | 首屏加载、下拉刷新、10 分钟轮询 |

### 6.2 API-01：今日个人统计

**请求**：

```
GET /api/tenant/stats/mine?dateFrom={dateFrom}&dateTo={dateTo}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `dateFrom` | string | 是 | 起始日期，格式 `YYYY-MM-DD`，取当前日期 |
| `dateTo` | string | 是 | 结束日期，格式 `YYYY-MM-DD`，取当前日期 |

**请求示例**：

```
GET /api/tenant/stats/mine?dateFrom=2026-07-23&dateTo=2026-07-23
Authorization: Bearer {accessToken}
```

**响应（200，统一信封，见 `00-全局API约定.md` §1）**：

```json
{
  "success": true,
  "data": {
    "myToday": {
      "followupCount": 12,
      "answeredCount": 8
    },
    "myLeadsTotal": 128,
    "...": "其他字段（myFollowed / myAnswered / myConverted / myConversionRate 等）"
  },
  "error": null
}
```

**前端字段映射**：

| UI 指标 | 取值路径 | 转换逻辑 |
|---------|---------|---------|
| 今日待办 | API-03 `data.byStatus.dueToday` | 直接显示（非本接口字段，见 §6.4） |
| 今日跟进 | `data.myToday.followupCount` | 直接显示 |
| 今日接通 | `data.myToday.answeredCount` | 直接显示 |
| 我的线索 | `data.myLeadsTotal` | 四宫格右下角 + 快捷入口副标题显示 |

> **v1.1 补充（REQ-07）**：「今日转化数 / 今日拨打量」待后端 `stats/mine` 扩展 `myToday.convertedCount / callCount` 后恢复展示，MVP 不渲染、不占位。

**错误响应**（失败信封 `{ "success": false, "data": null, "error": { "code", "message" } }`，错误码见 `00-全局API约定.md` §8）：

| HTTP 状态码 | 前端处理 |
|------------|---------|
| 401 | accessToken 过期 → 网络层自动用 refreshToken 换发并重试；refreshToken 也失效 → 跳转登录页 |
| 400 | `VALIDATION`（如漏传 `dateFrom`/`dateTo`）→ 属前端参数错误，不展示给用户，记录日志 |
| 423 | 强制改密 → 跳转强制改密页（详见 `APP_PRODUCT_DESIGN.md` §4.1.2） |
| 500 / 网络超时 | 显示错误态 + 重试按钮 |

### 6.3 API-02：待办日程预览

**请求**：

```
GET /api/tenant/schedules?status=pending&page=1&size=5&sort=scheduledAt
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `status` | string | 是 | 固定传 `pending`，仅取待办日程 |
| `page` | int | 是 | 固定传 `1`，首页仅取第一页 |
| `size` | int | 是 | 固定传 `5`，最多 5 条 |
| `sort` | string | 是 | 固定传 `scheduledAt`，按预定时间升序 |

**请求示例**：

```
GET /api/tenant/schedules?status=pending&page=1&size=5&sort=scheduledAt
Authorization: Bearer {accessToken}
```

**响应（200，统一信封 + 标准分页体，见 `00-全局API约定.md` §1/§2）**：

```json
{
  "success": true,
  "data": {
    "items": [
      {
        "id": "sch_001",
        "title": "回访张先生",
        "content": "确认产品意向",
        "scheduledAt": 1753252200,
        "status": "pending",
        "lead": {
          "name": "张明",
          "phone": "138****1234"
        },
        "createdAt": 1753165800
      }
    ],
    "total": 15,
    "page": 1,
    "size": 5,
    "pages": 3
  },
  "error": null
}
```

> 时间字段为 **int Unix 秒**（后端存储与传输格式），展示层统一格式化为 `YYYY-MM-DD HH:mm`（见 `00-全局API约定.md` §7）。`lead` 为关联线索快照，线索被擦除时为 `null`；TE/TM 视角 phone 脱敏，TA 明文。

**前端字段映射**：

| UI 元素 | 取值路径 | 说明 |
|---------|---------|------|
| 日程时间 | `item.scheduledAt` | Unix 秒，格式化为 "HH:mm"（取时分） |
| 日程标题 | `item.title` | 直接显示，`maxLines=1` 截断 |
| 线索姓名 | `item.lead?.name` | 若 `lead` 为 `null` 则显示 "无" |
| 是否逾期 | `item.scheduledAt < now && item.status === 'pending'` | 前端本地判断（now 为服务端时间 Unix 秒） |
| TDBadge 计数 | API-03 `data.byStatus.dueToday` | 今日待办数，显示在 Section 标题旁（不取本接口 `total`） |

**逾期判断逻辑**：

```
isOverdue = (schedule.scheduledAt < currentServerTime) && (schedule.status === 'pending')
```

> **注意**：逾期判断使用服务端时间而非设备本地时间，避免用户手动修改手机时间导致误判。服务端时间可从 HTTP 响应头 `Date` 获取并缓存。

### 6.4 API-03：我的日程统计

**请求**：

```
GET /api/tenant/schedules/stats/mine
Authorization: Bearer {accessToken}
```

**响应（200，统一信封）**：

```json
{
  "success": true,
  "data": {
    "byStatus": {
      "pending": 15,
      "completed": 42,
      "cancelled": 3,
      "overdue": 2,
      "dueToday": 5
    }
  },
  "error": null
}
```

**前端字段映射**：

| UI 元素 | 取值路径 | 说明 |
|---------|---------|------|
| 四宫格"今日待办" | `data.byStatus.dueToday` | 口径：`pending` 且 `scheduledAt ≤ 今日结束` |
| 待办日程 Section TDBadge | `data.byStatus.dueToday` | 与四宫格第一项同源同口径 |
| Tab 3「日程」Badge | `data.byStatus.dueToday` | 与主文档 §3.2 Tab Badge 定义一致 |

> **⚠ 后端依赖（REQ-05）**：`dueToday` 字段为后端需求 REQ-05 新增（见 `APP_BACKEND_REQUIREMENTS_REVIEW_SUPPLEMENT.md`，口径：`pending` 且 `scheduledAt ≤ 今日 23:59:59`，逾期待办 `overdue ⊆ dueToday`）。该字段上线前，前端以 `byStatus.pending` 兜底并标注口径差异；`overdue` 为 `pending` 且 `scheduledAt ≤ now` 的派生计数，不是独立日程状态（见 `00-全局API约定.md` §4.4）。

### 6.5 API-04：即将到期日程检测（日程到期提醒条）

**请求**（检测未来 30 分钟内到期的未完成日程，仅取计数）：

```
GET /api/tenant/schedules?status=pending&scheduledAt__gte={now}&scheduledAt__lte={now+1800}&page=1&size=1
Authorization: Bearer {accessToken}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `status` | string | 是 | 固定 `pending` |
| `scheduledAt__gte` | int | 是 | 当前服务端时间 Unix 秒（排除已逾期） |
| `scheduledAt__lte` | int | 是 | 当前服务端时间 + 1800 秒（未来 30 分钟内到期） |
| `page` / `size` | int | 是 | `1` / `1`，仅为取 `total` 计数，不消费列表数据 |

**响应（200，统一信封 + 标准分页体）**：

```json
{
  "success": true,
  "data": {
    "items": [ { "id": "sch_009", "title": "回访张先生", "scheduledAt": 1753253400, "status": "pending" } ],
    "total": 2,
    "page": 1,
    "size": 1,
    "pages": 2
  },
  "error": null
}
```

**前端字段映射**：

| UI 元素 | 取值路径 | 说明 |
|---------|---------|------|
| 提醒条 N | `data.total` | N ≥ 1 → 显示"您有 N 条日程即将到期"；N = 0 → 隐藏 |

> `now` 取服务端时间（HTTP 响应头 `Date`，与 §6.3 逾期判断同源），避免设备时间被篡改导致误报。过滤/排序参数遵循 `00-全局API约定.md` §3（驼峰字段 + `字段__op` 操作符）。

## 7. 边界情况与备注

### 7.1 数据边界

| 场景 | 处理方式 |
|------|---------|
| 统计数字为 0 | 正常显示 "0"，不隐藏卡片 |
| 统计数字 ≥ 1000 | 显示 "999+"，不显示完整数字 |
| 统计数字为负数 | 不应出现；若出现则显示 "0" 并上报异常日志 |
| 日程不足 5 条 | 有几条显示几条，不补齐占位 |
| 日程恰好 5 条 | 正常显示 5 条，"查看全部"仍可点击 |
| 日程标题为空字符串 | 显示 "未命名日程" |
| 日程标题超长 | 单行截断，末尾显示 "..."（`maxLines=1, overflow=TextOverflow.ellipsis`） |
| 关联线索已被擦除（`lead=null`） | 线索名显示 "无"，日程条目仍可点击查看详情，但详情页内拨号按钮不可用 |
| `myLeadsTotal` 为 0 | 快捷入口副标题显示 "0 条"，入口仍可点击（进入空态列表） |
| 提醒条检测 N = 0 | 提醒条不显示；已显示的在下一轮检测后滑出隐藏（全部处理完/全部过期） |
| 提醒条被用户关闭 | 当次会话（APP 本次前台运行期间）不再提示；下一轮轮询检测到新的即将到期日程（ID 集合变化）可重新出现 |
| 提醒条跨午夜 | 午夜后轮询按新的一天重新检测，"即将到期"窗口始终以当前服务端时间为基准 |

### 7.2 时间边界

| 场景 | 处理方式 |
|------|---------|
| 跨午夜 00:00 | 页面在前台跨过 00:00 时，自动重新请求当天数据（日期变更检测） |
| 用户手动修改设备时间 | 使用服务端时间做逾期判断，不受设备时间影响 |
| 时区问题 | 日期参数统一使用设备本地时区（`YYYY-MM-DD`），与后端约定一致 |

### 7.3 网络边界

| 场景 | 处理方式 |
|------|---------|
| 弱网（高延迟） | TDSkeleton 骨架屏持续展示直到接口返回；设置超时 15 秒，超时后走错误态 |
| 请求竞态（Race Condition） | 下拉刷新 / 轮询 / 重试可能并发触发同一接口。使用请求去重（同一时刻只保留最新一次请求，取消旧请求） |
| 后台回前台 | APP 从后台切回前台时，检查距上次数据更新时间是否 > 10 分钟，若是则立即触发一次刷新 |
| 网络恢复 | 离线提示条消失 → 自动触发数据刷新 → 刷新完成前 TDSkeleton 覆盖旧数据（或保留旧数据 + 顶部 loading 条） |

### 7.4 权限边界

| 场景 | 处理方式 |
|------|---------|
| TE 角色 | TDNavBar 隐藏"团队看板"按钮；日程仅展示个人数据 |
| TM/TA 角色 | TDNavBar 显示"团队看板"按钮；首页仍展示个人数据（团队数据在独立页面） |
| Token 即将过期（< 5 分钟） | 网络层自动刷新 token，用户无感知 |

### 7.5 性能考量

| 项目 | 策略 |
|------|------|
| 接口并发 | stats 和 schedules 并行请求，不串行等待 |
| 列表 diff | 日程列表更新使用 Flutter 框架 diff 机制避免全量刷新闪烁 |
| 图片加载 | 空态插画使用 SVG/Vector，无需网络加载 |
| 内存 | 首页数据量小（4 个数字 + 5 条日程），无内存压力 |
| 轮询省电 | 10 分钟间隔足够长，对电池影响可忽略；APP 进入后台时停止轮询（`WidgetsBindingObserver` 的 `didChangeAppLifecycleState` 中移除定时器） |
