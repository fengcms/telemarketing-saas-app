## 1. 产品概述

### 1.1 产品定位

电销坐席的**工作手机工具**——不是管理后台的移动版，而是坐席每天打开、用来完成「看线索 → 打电话 → 记结果 → 约下次」全流程的专用客户端。

### 1.2 目标用户

| 角色 | 英文 | APP 定位 |
|------|------|---------|
| **租户员工** | TE (Tenant Employee) | **核心用户**。自己的线索、拨号、跟进、日程、个人统计 |
| **租户经理** | TM (Tenant Manager) | **辅助用户**。仅查看团队数据（线索池、统计），不做管理操作 |
| **租户管理员** | TA (Tenant Admin) | **同上**。仅查看数据，管理功能在后台 Web 做 |

> 平台超管（PSA）不在 APP 覆盖范围，所有管理操作使用后台 Web。

### 1.3 核心价值

- **减少操作步骤**：从线索到拨号到填跟进，三步内完成
- **半自动回传**：拨号后自动读通话记录，减少手填
- **移动化工作**：坐席在外/在家也能看到今天的待办清单和绩效数据

### 1.4 非目标（Non-goals）

> 以下为 APP MVP **明确不做**的功能，列于此防止需求蔓延和 stakeholder 预期错位。

| 非目标 | 原因 / 替代方案 |
|--------|----------------|
| ❌ **不做 iOS 版本** | 所有坐席使用安卓设备，不开发 iOS 端 |
| ❌ **不做推送通知** | MVP 无推送基础设施；坐席通过下拉刷新 + 10 分钟自动轮询感知新任务 |
| ❌ **不做 VoIP/WebRTC/点击外呼** | 仅调系统拨号盘发起通话，不集成外呼网关；通话过程中 APP 不参与（进入后台） |
| ❌ **不做管理后台功能** | 建用户/分类/项目/导入数据等管理操作使用桌面端 Web 后台。APP 中 TA/TM 仅查看，不做管理操作 |
| ❌ **不做实时协作/协同编辑** | 线索分配、日程变更等不实时推送；坐席通过查询最新数据感知变更 |
| ❌ **不做录音采集与播放** | `recording_url` MVP 恒 NULL，录音采集仅外呼网关集成后支持 |
| ❌ **不做跨租户/平台管理** | APP 仅消费所在租户数据，平台超管（PSA）不使用 APP |
| ❌ **不做离线模式** | APP 需要网络连接；离线时展示离线提示，仅本地 draft 持久化可用 |
| ❌ **不做应用商店发布** | 企业侧载分发，直接从下载链接安装 APK；不经过 Google Play 审核 |
| ❌ **不做多语言国际化** | MVP 仅中文 UI |
| ❌ **不代收发邮件/短信** | APP 不集成邮件客户端或短信网关，与客户的沟通通过电话完成 |

### 3.2 底部导航栏（4 Tab）

| Tab | 图标 | 名称 | TE | TM/TA | Badge 显示 |
|-----|------|------|:--:|:-----:|-----------|
| 1 | 🏠 | 首页 | 个人看板 | 个人看板 | — |
| 2 | 📋 | 线索 | 我的线索 | 全部线索 | — |
| 3 | 📅 | 日程 | 我的日程 | 我的 + 团队日程 | 今日待办数（`pending` 且 `scheduledAt ≤ 今日结束`） |
| 4 | 👤 | 我的 | 个人中心 | 个人中心 + 团队入口 | — |

> Badge 数据通过与首页同步的 10 分钟轮询获取，仅显示数字，不设红点。

### 3.3 全局交互规范

| 规范 | 说明 |
|------|------|
| **下拉刷新** | 所有列表页支持下拉刷新（后端无 WebSocket/SSE，前端手动刷新） |
| **自动轮询** | APP 前台运行时每 **10 分钟**自动拉取一次首页统计数据（跟进数、待办数），更新 Tab Badge。不频繁请求接口 |
| **无限滚动** | 列表统一走后端 `page/size` 分页，默认 `size=20`，上拉加载更多 |
| **空状态** | 列表无数据时展示友好空态插画 + 提示文案 |
| **加载态** | 列表首屏加载用 Skeleton，后续加载用底部 loading indicator |
| **错误态** | 网络错误统一展示重试按钮 + 错误描述 |

> **关于无推送通知的说明**：后端 MVP 不提供推送通知服务，线索分配/日程变更等不会实时推送到 APP。
> 坐席通过以下方式感知新任务：
> - **手动下拉刷新**线索/日程列表
> - **10 分钟自动轮询**更新 Badge 数值
> - 经理或管理员在线下/后台告知

## 4. 功能模块详细设计

### 4.1 认证模块

#### 4.1.1 登录

| 项目 | 说明 |
|------|------|
| 入口 | APP 启动 → 未登录 / Token 过期 → 登录页 |
| 输入 | 邮箱 + 密码 |
| 流程 | ① POST `/api/auth/login` → ② 收到 `{accessToken, refreshToken, user}` → ③ 检查 `user.mustResetPassword` → ④ 若为 1 则跳转强制改密页；若为 0 则进首页 |
| 错误处理 | `401` → "邮箱或密码错误"；`423 ACCOUNT_LOCKED` → "账号已锁定，请 15 分钟后再试"；其他错误码见附录 B |
| Token 管理 | accessToken（1h）过期前自动用 refreshToken（7d）换发；refreshToken 过期 → 回到登录页 |

> **强制改密检测**：使用登录响应中的 `user.mustResetPassword` 字段而非等待 423 错误码。
> 因为后端登录接口返回 200（正常 Token 但内含 mrp=1），423 仅在后端受保护路由的 `authMiddleware` 中触发。

#### 4.1.2 强制改密

| 项目 | 说明 |
|------|------|
| 触发条件 | 管理员重置密码后 `user.mustResetPassword=1`，登录成功后直接跳转改密页 |
| 流程 | ① 新密码输入（最少 8 位）→ ② `POST /api/auth/change-password`（无 `oldPassword`）→ ③ 成功后后端自增 `tv` 使旧 Token 全部失效 → ④ APP 清空本地 Token → ⑤ 回到登录页，用新密码重新登录 → ⑥ 获得新 Token（mrp=0）正常使用 |
| 全局拦截 | 未改密前任何 API 请求均被后端 authMiddleware 返回 `423 FORCE_CHANGE_PASSWORD`。APP 网络层拦截器应捕获此状态码：**不走 refresh→retry 路径**，直接跳转至改密页。避免死循环：刷新 token（mrp 仍为 1）→ 重试原请求（再 423）→ 刷新 → … |
| 逃生路径 | 登录响应中 `mustResetPassword=1` 是首选检测方式（不触发 API 请求就跳转）；`423 FORCE_CHANGE_PASSWORD` 是兜底防护（防止其他页面 init 请求触发重定向循环） |

#### 4.1.3 登出

| 操作 | 接口 | 本地行为 |
|------|------|---------|
| 单设备登出 | POST `/api/auth/logout`（传 refreshToken） | 清空本地 Token → 回登录页 |
| 全设备登出（可选功能） | POST `/api/auth/logout-all` | 清空本地 Token → 回登录页 |

> 登出后通知用户：其他设备若使用中也会因 tv 自增而失效（logout-all 场景）。

### 4.2 首页看板

#### 4.2.1 页面结构

顶部分两行：
- **今日概况卡片**：今日待办数、今日跟进数、今日接通数、我的线索数
- **待办日程预览**：最多 5 条今天的日程，点击进入完整日程列表

#### 4.2.2 数据来源

| 数据项 | API | 说明 |
|--------|-----|------|
| 今日待办数 | `GET /api/tenant/schedules/stats/mine` → `dueToday` | `pending` 且 `scheduledAt ≤ 今日结束`（后端 REQ-05 新增字段） |

| 我的线索总数 | `GET /api/tenant/stats/mine` → `myLeadsTotal` | 首页不需要列表，仅显示计数 |

#### 4.2.3 刷新策略

- 进入页面时自动拉取
- 支持下拉刷新
- **前台每 10 分钟自动轮询**更新统计和 Badge
- TM/TA 首页右上角可加「团队看板」入口按钮

### 4.3 待办日程

#### 4.3.1 日程列表

| 项目 | 说明 |
|------|------|
| 入口 | 底部 Tab 3「日程」 |
| TE 数据范围 | 仅自己的日程（`user_id=me`），后端默认过滤 |
| TM/TA 数据范围 | 可切换"我的" / "团队全部" |
| 列表排序 | 默认按 `scheduledAt` 升序，最紧急的排最前 |
| 过期标记 | `scheduledAt < now` 且 `status=pending` → 卡片标红色/橙色 + "已逾期"标签 |
| 筛选 | `?status__in=pending,completed` 切换「待办」/「已完成」 |
| 列表字段 | 标题、关联线索姓名+电话（**不脱敏**）、预定时间、状态标签 |
| 搜索 | `q` 参数搜索 `schedules.title` + `schedules.content` + `leads.name` |

> **日程列表 phone 不脱敏的原因**：日程列表通过 LEFT JOIN leads 直接返回完整的电话号码，因为日程是坐席自己的待办事项，数量有限（仅 pending + 当日），不会形成批量导出风险。后端约定如此，APP 直接展示。
>
> **关于已擦除线索**：若关联线索已被 TA 通过 PIPL 擦除（PII 置空），日程列表中的 `lead` 字段为 `null`，但日程条目本身不受影响（标题、时间、状态仍正常展示）。此场景极少出现（仅当管理员擦除线索时该线索仍有待办日程），APP 无需特殊处理，展示无关联提示即可。

#### 4.3.2 日程详情与操作

点击日程进入详情，展示：

- 日程标题、内容
- 关联线索（姓名、电话、项目）
- 创建时间、计划时间
- 操作按钮（根据状态）：

| 当前状态 | 可操作 |
|---------|--------|
| pending（未过期） | 完成 / 取消 / 编辑时间 / **删除** / **📞 拨号**（跳转关联线索详情页，自动聚焦拨号按钮） |
| pending（已过期） | 完成 / 取消 / 编辑时间 / **跳转线索拨号** / **删除** |
| completed | 重新打开（reopen → pending） |
| cancelled | 重新打开（reopen → pending） |

> **拨号按钮说明**：所有 `pending` 状态的日程（无论是否过期）均可一键跳转到关联线索的详情页并聚焦拨号按钮，减少操作步骤。如果关联线索已被擦除（`lead` 字段为 `null`），拨号按钮不可用并提示"线索已被删除"。

**日程到期提醒**：

APP 处于前台时，每 **10 分钟自动轮询** 检查一次是否有未来 **30 分钟内** 到期的待办日程。若有，在首页顶部展示非侵入式提示条：
- 提示内容："您有 N 条日程即将到期"
- 点击提示条 → 跳转日程列表（自动过滤 `pending` + 筛选 `scheduledAt ≤ 当前时间+30min`）
- 关闭提示条 → 标记已读，不再重复提示

**操作 API 映射**：
- ✅ 完成 → `POST /api/tenant/schedules/:id/complete`
- ❌ 取消 → `POST /api/tenant/schedules/:id/cancel`
- 🔄 重新打开 → `POST /api/tenant/schedules/:id/reopen`
- ✏️ 改期 → `PATCH /api/tenant/schedules/:id`（`{scheduledAt,title,content}`）
- 🗑 删除 → `DELETE /api/tenant/schedules/:id`（本人可删自己的，TA 可删全部）

#### 4.3.3 新建日程

| 项目 | 说明 |
|------|------|
| 入口 | 线索详情 → "预约下次跟进" 按钮；或跟进提交后可选 |
| 场景 | 打完电话后，约下次沟通时间 |
| TE 限制 | 只能给自己归属的线索建日程（后端校验） |
| TM/TA 限制 | 可替任意线索建，可选传 `userId` 指定归属员工 |
| 必填 | `leadId`（当前线索）、`scheduledAt`（下次跟进时间） |
| 可选 | `callRecordId`（关联本次通话记录）、`title`、`content`、`userId`（TM/TA 替员工建） |
| 联动 | 建后联动更新 `leads.nextFollowupAt`（后端自动完成） |

### 4.4 线索管理

#### 4.4.1 线索列表

| 项目 | 说明 |
|------|------|
| 入口 | 底部 Tab 2「线索」 |
| TE 范围 | 自动 `scope=mine`（自己的线索），排除 `converted`、`is_blocked=1`、**已擦除记录** |
| TM/TA 范围 | 自动 `scope=all`（团队全部），排除同上 |
| 列表展示 | 姓名（脱敏）、电话（脱敏 `138****1234`）、项目、分类、状态标签、最后跟进时间、**最后跟进摘要**（截取最近一次跟进备注前 20 字 + 接听类型标签）、**下次跟进时间标识**（`nextFollowupAt` 倒计时徽章：绿色=今日可打、橙色=未来时间） |
| 排序 | 默认按 `updatedAt` 降序（最近跟进排最前）；**筛选面板中可选**按 `nextFollowupAt` 升序（待跟进优先排最前） |
| 搜索 | `q` 参数全局搜索姓名/电话/公司 |
| 搜索交互 | 线索列表顶部搜索框，支持**实时输入搜索**（防抖 500ms 后自动发起请求）和**回车确认搜索**两种触发方式；搜索范围为线索的姓名/电话/公司字段，后端使用 LIKE 模糊匹配；搜索结果替换当前列表内容，搜索后保留筛选/排序条件；清除搜索文本后恢复原始列表 |
| 筛选面板 | 支持按状态、分类、项目、时间段筛选（使用通用查询 DSL `字段__操作符=值`）；筛选条件与搜索可叠加使用 |
| 关键参数 | **所有调用均携带 `erased=0`**，排除已擦除的「空记录」，避免列表出现全 null 的脏条目 |
| 空态 | 无线索 → "暂无线索，请联系管理员导入" |

#### 4.4.2 线索详情

页面分区：

**A. 头部信息区**

- 状态标签 + 分类标签

**B. 操作区（核心）**

### 4.5 拨号与回传

这是 APP **最核心的流程**，拨号→挂断→回传→填跟进，要求交互流畅、步骤少。

#### 4.5.1 完整流程

```
┌──────────────────────────────────────────────────────────┐
│ ① 用户按下拨号按钮                                        │
│    → 检查夜间禁呼时段（若命中则弹确认框）                     │
│    → 用户确认后，App 调用系统拨号盘（Intent.ACTION_DIAL）    │
│    → Android 将通话切换到系统拨号界面                       │
│    → App 进入后台（**拨号时即预生成 UUID** 存本地 draft）    │
├──────────────────────────────────────────────────────────┤
│ ② 用户通话（发生在手机基带，App 不参与）                     │
│    → 通话结束后，用户主动切回 APP                           │
├──────────────────────────────────────────────────────────┤
│ ③ APP 检测到从后台返回前台                                  │
│    → 自动读取 READ_CALL_LOG 获取最近通话记录                  │
│    → 匹配拨打号码 + 通话时间                                  │
│    → 自动填充：startedAt、endedAt、duration                 │
│    → 弹出「通话结果反馈面板」（模态/半屏）                      │
├──────────────────────────────────────────────────────────┤
│ ④ 员工在反馈面板中填写：                                     │
│    □ 接听类型  [已接听/无人接听/拒接/空号/停机]（必选）       │
│    □ 跟进备注  [自由文本 / 选择快捷备注]（可选）               │
│    □ 线索分类  [选择分类]（可选，当前分类可改）                │
│    □ 下次跟进时间（可选）→ 触发新建日程                        │
│    ▶ 点击「提交」                                           │
├──────────────────────────────────────────────────────────┤
│ ⑤ App 发起单接口提交（原子创建通话+跟进）：                    │
│    POST /api/tenant/calls ── 携 externalCallId（幂等）+      │
│      answerType + content（跟进备注）+ categoryId（可选）     │
│    → 后端在同一事务内原子创建「通话记录 + 跟进记录」，          │
│      并同步线索 status→following、lastFollowupAt              │
│    → 日程不在此事务内：需要时由 12-新建日程页独立创建           │
│      （POST /api/tenant/schedules，可携 callRecordId 关联）    │
│    → 成功后关闭面板，清除本地 draft，刷新线索详情页时间线         │
└──────────────────────────────────────────────────────────┘
```

#### 4.5.2 夜间禁呼软提醒（noCallWindow）

| 项目 | 说明 |
|------|------|
| 数据来源 | `GET /api/tenant/profile` → `response.settings.noCallWindow`（租户配置，全部角色可读）<br>APP 启动后或首次进入首页时拉取一次，缓存到本地，不频繁请求 |
| 配置结构 | `{ enabled: bool, start: "21:00", end: "09:00" }`（租户级配置） |
| 触发时机 | 用户点击拨号按钮时，但 API 请求尚未发起前 |
| UI 表现 | 若当前时间在禁呼窗口内，弹出确认对话框：<br>`"当前时间（21:30）在禁呼时段（21:00~09:00）内，确定要拨打吗？"`<br>按钮：「取消」「确定拨打」|
| 用户确认后 | 正常调系统拨号盘；后端 `POST /calls` 会根据 `startedAt` 自动打 `violation=1` 标记 |
| 用户取消 | 不调拨号盘，返回线索详情页 |
| 无法硬阻断 | 因为调用的是系统拨号盘，APP 无法在通话前真正阻止拨号。此软提醒是后端 `violation` 标记之外的合规层面保护 |
| MVP 策略 | 若接口支持读取 `noCallWindow` 则做；若暂不支持，跳过此功能，不影响拨号流程 |

#### 4.5.3 自动回传技术要点

| 要点 | 说明 |
|------|------|
| 权限 | 首次安装打开时申请 `READ_CALL_LOG`（用户允许后才能在挂断后自动读取） |
| 匹配策略 | 按拨出的号码 + 通话开始时间（最近 2 分钟内）匹配系统通话记录 |
| 匹配失败 | 未读到/权限拒绝 → 反馈面板仍弹出，`startedAt`/`duration`/`answerType` 让用户手动填写 |
| 幂等 | `POST /api/tenant/calls` 必须携带 `externalCallId`（App 生成 UUID），后端已建立唯一索引确保同一通通话不会重复创建 |

#### 4.5.5 未提交通话的持久化管理

| 场景 | 行为 |
|------|------|
| 用户关闭面板没提交 | 线索详情页底部显示「有 1 条未提交通话记录」提示条，点击重新打开面板 |
| 用户打了多个电话都没提交 | 按通话时间顺序，在最近一条未提交的线索上提示 |
| 切换到其他线索 | 未提交记录保留在当前线索，切换回来仍可见提示 |
| **APP 被切换到后台或被系统杀死** | 未提交记录持久化到本地存储（最小值集）：`{leadId, phone, externalCallId, startedAt, endedAt?, duration?, draft: true}`。重启 APP 后检测到 draft 状态的记录，在相应线索详情页显示提示条 |
| 提交成功后 | 清除本地对应的 draft 记录 |

> **持久化策略**：使用本地 JSON 文件或 SharedPreferences 保存 draft 数据集，不要求 SQLite。仅保存最小必要字段。点击拨号时即生成 `externalCallId`（UUID）并写入 draft，确保即便 APP 崩溃后也能通过幂等机制正确更新历史通话。

#### 4.6.2 跟进记录详情与操作

点击时间线中的某条跟进，展开详情（或底部 sheet）：

**跟进记录操作（TE 权限）**：

| 操作 | 条件 | 接口 | 说明 |
|------|------|------|------|

> **✅ 后端已实现（2026-07-22）**：`PATCH /api/tenant/leads/:id/followups/:fid` 和 `DELETE /api/tenant/leads/:id/followups/:fid` 均已上线。迁移 0012 为 `lead_followups` 新增 `deleted_at` 列支持软删。TE 5 分钟编辑窗口、权限校验、审计记录均已按设计实现。APP 端可直接对接。

> **设计理由**：TE 写错跟进备注是高频场景（每天都会发生），当前无任何修改入口。允许 5 分钟内编辑备注可覆盖「提交后发现写错」的典型场景，又不至于无限期修改导致审计混乱。TM/TA 不受限制，可做事后纠错。

### 4.9 统计看板

#### 4.9.1 个人统计（TE）

`GET /api/tenant/stats/mine?dateFrom=YYYY-MM-DD&dateTo=YYYY-MM-DD` 真实响应结构：

| 字段 | 类型 | 说明 |
|------|------|------|
| `myLeadsTotal` | Number | 当前归属我的线索总量 |
| `myFollowed` | Number | 累计跟进数 |
| `myAnswered` | Number | 累计接通数 |
| `myNoAnswer` | Number | 累计未接数 |
| `myConverted` | Number | 累计转化数 |
| `myConversionRate` | Number | 转化率百分比（如 12.5） |
| `myToday.followupCount` | Number | **今日**新增跟进数 |
| `myToday.answeredCount` | Number | **今日**接通电话数 |

> 注意：后端 `stats/mine` **不返回** `myByStatus`/`myByCategory`/`myByAnswerType` 等分组字段。
> 按状态/分类/接听类型的分布数据仅在团队级统计（`GET /api/tenant/stats`）中可用。
> `dateFrom` 和 `dateTo` 为必传参数。
> **转化率口径（产品已确认）**：`myConversionRate = myConverted / myLeadsTotal`（转化数 ÷ 名下线索总数）。分母为 0 时返回 0。
> ⚠️ 团队统计 `agentPerf[]` **不返回** `conversionRate` 字段，前端按同口径自算：`convertedCount / ownedLeads`。

#### 4.9.2 团队统计（TM/TA）

`GET /api/tenant/stats?dateFrom=YYYY-MM-DD&dateTo=YYYY-MM-DD` 完整响应字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `total` | Number | 团队活跃线索总数 |
| `byStatus` | Object | 各状态分布：`pending`/`assigned`/`following`/`converted`/`invalid` |
| `byCategory` | Object | 按分类 ID 分组的线索数 |
| `byProject` | Object | 按项目 ID 分组：`{count, converted}`/项目 |
| `poolTotal` | Number | 公海线索数 |
| `staleInPool` | Number | 公海中超 7 天未分配的呆滞线索数 |
| `funnel` | Object | 转化漏斗：`{pool, assigned, following, converted}` |
| `agentPerf[]` | Array | 坐席绩效排行：`{userId, name, ownedLeads, followedCount, answeredCount, noAnswerCount, convertedCount, avgDuration, lastFollowupAt}`（无 `conversionRate`，前端按 `convertedCount/ownedLeads` 自算） |
| `byAnswerType` | Object | 接听类型分布：`{answered, no_answer, rejected, empty_number, suspended}`（后三项 MVP 简化版恒为 0） |
| `dailyTrend[]` | Array | 逐日趋势：`{date, added, followup, converted, answered, total}` |
| `compareYesterday` | Object | 较昨日变化：`{addedDiff, followupDiff, convertedDiff}` |

> 所有统计均需 `dateFrom`/`dateTo` 参数。APP 首版可仅展示核心指标（total, byStatus, agentPerf, funnel），其余字段保留接口能力后续扩展。

## 6. 数据流详解

### 6.1 核心链路：拨号→回传→跟进→日程

```
用户操作                    APP 行为                             后端
─────────                 ──────────                          ──────
① 点击号码                检查 noCallWindow (可选)
                           ↓
                          调 Intent.ACTION_DIAL
                          生成 externalCallId + 写本地 draft
② 通话（系统拨号盘）        App 后台                            不参与
③ 返回 APP                检测前台恢复
                           ↓
                          读 READ_CALL_LOG 匹配最近通话
                           ↓
                          填充 draft: startedAt, endedAt, duration
                          弹出反馈面板
④ 填写反馈面板             用户选 answerType + 填备注 + 选分类
                           ↓
                          点击提交
                           ↓
                          单接口原子提交：
                          POST /calls（externalCallId 幂等，携 content/categoryId）
                          → 后端同事务写入 call_records + lead_followups，
                            并同步 leads.status→following
                          （日程独立：12-新建日程页 POST /schedules，可携 callRecordId）
                           ↓
                          清除本地 draft
                          刷新线索详情页的时间线
```

### 6.2 Token 生命周期

```
首次登录 (200, user.mustResetPassword)
  │
  ├─ mustResetPassword===1 → 强制改密页
  └─ mustResetPassword===0 → 首页
  │
  ├─ 正常使用
  │   ├─ 每次请求: 检查 accessToken 是否过期
  │   ├─ 未过期 → 直接使用
  │   └─ 已过期 → POST /refresh 换发 → 重试原请求
  │
  ├─ 收到 423 FORCE_CHANGE_PASSWORD (其他页面 init 请求触发)
  │   └─ 不走 refresh 路径，直接跳转改密页
  │
  ├─ 收到 401 AUTH_EXPIRED (jti 吊销/tv 不匹配)
  │   └─ POST /refresh → 若 refresh 也失效 → 回登录页
  │
  ├─ 登出
  │   └─ POST /logout → 清本地 Token → 回登录页
  │
  └─ refreshToken 过期
      └─ 回登录页
```

### 6.3 幂等机制

```
场景: 通话结果回传时，用户可能多次点击「提交」
或网络闪断后重试 → 相同通话被 POST 两次

防御: POST /calls 的 externalCallId 字段
  - APP 生成 UUID（在点击拨号时即生成，存储本地 draft）
  - 后端 UNIQUE(tenant_id, external_call_id)
  - 已存在 → 更新原记录（补正时长/听类型）
  - 不会重复创建两条

注意: 跟进和日程没有显式幂等字段
  - APP 在提交时做前端防重复点（按钮 loading + 禁用）
  - 断网重试场景由 APP 自行判定避免重复提交
```

## 附录 B：全局错误码处理策略

APP 网络层应统一拦截以下错误码并做出对应处理：

| 错误码 | HTTP | 用户提示 | APP 处理逻辑 |
|--------|------|---------|-------------|
| `AUTH_INVALID` | 401 | "邮箱或密码错误" | 登录页显示，不清除已有 Token |
| `AUTH_EXPIRED` | 401 | "登录已过期，请重新登录" | 尝试 POST /refresh；若 refresh 也失效 → 清 Token → 回登录页 |
| `AUTH_FORBIDDEN` | 403 | "没有权限执行此操作" | 弹窗提示后返回上一页 |
| `TENANT_EXPIRED` | 403 | "您的租户已过期，请联系管理员" | 弹窗 → 跳登录页，阻止进一步操作 |
| `TENANT_IN_GRACE` | 403 | "租户已到期，仅可查看数据" | 展示只读提示条，禁用所有写操作入口 |
| `TENANT_SUSPENDED` | 403 | "租户已被暂停，请联系管理员" | 弹窗 → 跳登录页 |
| `RATE_LIMITED` | 429 | "操作太频繁，请稍后再试" | 弹窗提示；静默重试（2 秒后自动重试一次） |
| `ACCOUNT_LOCKED` | 423 | "账号已锁定，请 15 分钟后再试" | 登录页显示，不可操作 |
| `FORCE_CHANGE_PASSWORD` | 423 | "需要修改密码后才能继续使用" | 不走 refresh 路径，直接跳转强制改密页 |
| `VALIDATION` | 400 | 根据 `message` 字段展示具体提示 | 对应输入框显示校验错误 |
| `NOT_FOUND` | 404 | "未找到该记录，可能已被删除" | 弹窗提示，返回上一页 |
| `BLOCKLIST_REJECTED` | 422 | "该手机号在禁拨名单中，无法拨号。联系管理员处理" | 拨号或跟进提交流程中 phone 校验不通过时显示 |
| `QUOTA_EXCEEDED` | 422 | "已超过配额上限" | 弹窗提示，不触发自动重试 |
| `BATCH_TOO_LARGE` | 413 | "批量操作数量超过上限" | 弹窗提示，建议减少选择数量后重试 |
| `INTERNAL` | 500 | "系统异常，请稍后再试" | 弹窗提示，不触发自动重试 |

> 未在此列表的错误码统一弹窗提示"系统异常（错误码：XXX），请联系技术支持"。

### 重试与请求队列策略

| 场景 | 策略 | 说明 |
|------|------|------|
| **网络闪断（HTTP 0 / timeout）** | APP 网络层自动重试**1 次**（2 秒延迟） | 不弹窗，静默重试；重试仍失败则弹"网络连接失败，请检查网络后重试" |
| **429 RATE_LIMITED** | 自动重试 **1 次**（2 秒延迟） | 弹窗提示"操作太频繁"，同时静默重试一次；不再接受人工反复点击 |
| **500+ / INTERNAL** | **不自动重试** | 弹窗提示"系统异常，请稍后再试"，引导用户手动重试 |
| **423 FORCE_CHANGE_PASSWORD** | **不走 refresh 路径** | 网络拦截器捕获此状态码时直接跳转强制改密页，避免 refresh → retry 死循环 |
| **401 AUTH_EXPIRED** | 自动 refresh 后重试原请求（最多 **1 轮**） | refresh 成功 → 用新 token 重试；refresh 也失败 → 清 Token → 回登录页 |
| **网络恢复后重试** | **不支持自动重试队列** | 断网期间未提交的请求（跟进/日程/通话回传）不会被自动重放，APP 仅在**提交入口**保留提示，用户需手动重新提交。draft 数据在本地持久化，不会丢失 |
| **用户主动重试** | 错误态 UI 统一提供「重试」按钮 | 所有错误态（网络断开、加载失败）展示重试按钮，用户点击后重新发起完整请求流程 |

## 附录 D：MVP 降级场景与兜底说明

> APP MVP 阶段存在一些已知的能力缺口和降级策略，列于此用于客服培训材料、用户指南参考和工程团队规划后续迭代。

### D.1 无推送通知

**缺口**：APP 不集成推送通知服务（无 FCM / 自建推送通道），线索分配、日程变更、分配提醒等不会实时推送到 APP。

**降级策略**：
- 坐席通过**手动下拉刷新**线索/日程列表获取最新数据
- 底部 Tab Badge 每 **10 分钟自动轮询**更新数值
- 经理或管理员在线下/其他渠道（微信/企业通讯工具）告知坐席变更

### D.2 无录音采集

**缺口**：`call_records.recording_url` MVP 恒 NULL。系统通话记录不含录音文件，外呼网关集成后方能回传录音。

**降级策略**：
- APP 不展示录音入口和播放控件
- 外呼网关集成前，通话质量评估依赖纯数据（接听类型、时长）
- 录音相关接口（PATCH recordingUrl）已预留，网关接入后 APP 无需改接口逻辑

### D.3 纯手动拨号，无点击外呼

**缺口**：APP 调系统拨号盘，而非点击即呼叫。坐席需要手动点击拨号、通话结束后切回 APP。

**降级策略**：
- APP 核心交互设计围绕"拨号 → 切出 → 切回 → 半自动回传"闭环，流程成熟
- 夜间禁呼为"软提醒"而非硬阻断（无法在系统拨号盘层面阻止拨号）
- 后端通过 `violation` 标记在通话记录层面做合规追述

### D.4 无后端实时数据订阅

**缺口**：后端不支持 WebSocket / SSE 等实时推送。

**降级策略**：
- 所有列表页支持下拉刷新
- 前台 10 分钟自动轮询
- 提交操作（跟进、日程操作）成功后自动刷新当前页面

### D.5 无离线使用能力

**缺口**：APP 大部分功能需要网络连接。离线时无法拉取线索/日程列表、无法提交数据。

**降级策略**：
- 网络中断时统一展示离线提示条（非阻断式，不弹窗）
- **未提交通话 draft 持久化**到本地存储（SharedPreferences/JSON），APP 崩溃或重启后仍可恢复
- 用户可从 draft 继续补充接听类型和备注后提交
- 不支持操作队列排队（网络恢复后不会自动重试未提交请求，需要手动操作）

### D.6 无应用商店分发

**缺口**：APP 不经 Google Play 上架，通过企业侧载链接分发 APK。

**降级策略**：
- 首次安装时风险提示：告知用户此为企业内部应用，来自可信来源
- 版本更新通过**后端 `/api/tenant/profile` `settings.appVersion` 字段**（预留）或线下分发的 APK 链接通知，不集成应用内自动升级 SDK
- 侧载分发不受应用商店审核约束，可自由申请 `READ_CALL_LOG` 等敏感权限

### D.7 无 iOS 版本

**缺口**：仅安卓端可用。

**降级策略**：
- 本文档仅针对安卓 APP 设计，不包含 iOS 端需求
- 所有坐席统一使用安卓设备
