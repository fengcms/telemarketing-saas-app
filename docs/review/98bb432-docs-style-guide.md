## 三、组长备注（对规范本身的反馈）

该规范是后续所有提交的对标基准，整体质量高，仅提两点供规范本身迭代参考：

1. **§1.3「方法必须有 `///`」是否区分私有辅助方法？** 当前页面普遍把 `build()` 拆成十几个 `_buildXxx()` 私有辅助方法，逐一起草 `///` 成本高且价值有限。建议明确：「公开类 / 公开方法 / 复杂私有方法」必须注释，「简单私有 UI 辅助方法」可不注。否则规范与现实脱节，难以落地。
2. **§3 常量命名**：规定「建议加 `k` 前缀」，但项目实际大量使用 `_keyXxx` / `LeadConstants.xxx` 风格。建议二选一写死，避免「建议」带来的不一致（当前 `token_storage`、`local_storage_service` 均为 `_keyXxx`）。

> 以上仅针对规范文档本身，不影响本提交「通过」。
